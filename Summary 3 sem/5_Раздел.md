## 5.1. Структуры данных для представления ориентированного графа с взвешенными дугами

По учебнику (раздел 13.1): **вершины графа** представляют объекты, **дуги (рёбра)** — связи между ними; у дуги может быть **вес** (стоимость, длина). **Каноническим** способом представления графа в учебниках является **матрица смежности или матрица инциденций**: наличие или отсутствие дуги между вершинами i и j отмечается ненулевым элементом; для взвешенного графа — собственно вес дуги (учебник 13.1).

**Ограничения матричного представления (по учебнику 13.1).** На практике матрица часто неэффективна. **(1) Разреженность графов:** большинство реальных графов **разреженные** — дуг мало по сравнению с максимально возможным числом; матрица заполнена преимущественно нулями, что ведёт к нерациональному использованию памяти. **Использование матриц оправдано только для полных или почти полных графов** (учебник). **(2) Множественность атрибутов:** дуги и вершины часто обладают многими атрибутами (длина, стоимость, время, тип); хранение в матрице проблематично. **(3)** Матрица требует сплошной нумерации вершин 0…N−1; в реальных системах объекты приходят из БД со своими идентификаторами — нужен механизм сопоставления. **(4)** Многие алгоритмы опираются на **окрестность** (множество входящих или исходящих дуг); из матрицы такую информацию извлекать неудобно (учебник 13.1).

**Представления для разреженных графов (по учебнику 13.1).** Целесообразно использовать **реализацию словаря**: лучше всего **хеш-таблица**, в которой **ключ — упорядоченная пара** вершин (i, j) для ориентированного графа; значение — вес или структура дуги. Альтернатива — **представление в виде списка дуг**: каждая дуга — объект с атрибутами; у **узла (Node)** — коллекция **исходящих дуг** (указатели или идентификаторы). Тогда перебор исходящих из вершины дуг эффективен; память O(V+E). В алгоритмической практике это часто реализуют как **список смежности**: для каждой вершины — список пар (конечная вершина, вес).

**Матрица смежности (классическая).** Двумерный массив adj[i][j] = вес дуги i→j (или метка «нет дуги»). Память O(V²), проверка и доступ к весу — O(1). Удобна для плотных графов и когда вершины уже пронумерованы.

```cpp
// Ориентированный граф с весами: список смежности (коллекция исходящих дуг у узла)
using Graph = std::vector<std::vector<std::pair<int, double>>>;  // (куда, вес)
Graph g(n);
g[u].push_back({v, w});  // дуга u → v с весом w

// Матрица смежности (по учебнику — каноническое представление; оправдана для плотных графов)
std::vector<std::vector<double>> adj(n, std::vector<double>(n, INF));
adj[u][v] = w;
```

**Итог 5.1.** По учебнику 13.1: **матрица смежности** — каноническое представление, но на практике часто неэффективна из‑за разреженности и множественности атрибутов; оправдана для полных или почти полных графов. Для разреженных графов предпочтительны **словарь** (ключ — упорядоченная пара вершин) или **список дуг / узел с коллекцией исходящих дуг** (список смежности): память O(V+E), эффективный перебор окрестности.

## 5.2. Структуры данных для представления неориентированного графа с взвешенными рёбрами

По учебнику (раздел 13.1): для целей представления можно рассматривать ориентированный и неориентированный граф единообразно; различие — в способе хранения дуг. В **неориентированном** графе ребро (u, v) связывает две вершины без направления; вес один для обоих концов. **Ключ в словаре** для неориентированного графа — **неупорядоченная пара** вершин (учебник 13.1: «ключом будет пара – упорядоченная в случае ориентированного графа и неупорядоченная в случае неориентированного»).

**Список смежности (представление через узлы с коллекцией дуг).** Ребро (u, v) с весом w храним **дважды**: в списке исходящих дуг вершины u — пара (v, w), в списке вершины v — пара (u, w). Так при обходе соседей любой вершины видны все инцидентные рёбра. Память O(V+E). Перебор соседей u — O(deg(u)).

**Матрица смежности.** **Симметричная:** adj[u][v] = adj[v][u] = вес ребра (u,v). Память O(V²), проверка ребра и доступ к весу — O(1). Те же ограничения, что и для ориентированного случая: оправдана для плотных графов (учебник 13.1).

**Итог 5.2.** По учебнику 13.1 неориентированный взвешенный граф представляют теми же структурами; в словаре ключ — **неупорядоченная пара** вершин. В списке смежности каждое ребро (u,v) хранится в коллекциях u и v; матрица смежности симметрична: adj[u][v] = adj[v][u].

## 5.3. Способы реализации матрицы смежности. Обзор и сравнительная характеристика

По учебнику (раздел 13.1): каноническим способом представления графа является **матрица смежности**; на практике различают **физическое** представление обычным двумерным массивом и **логическое** представление «матрицы» через другие структуры, когда граф разрежен.

**Плотная реализация (двумерный массив).** Матрица смежности хранится как **непрерывный массив размера V×V**: элемент adj[i][j] — вес дуги i→j (или признак отсутствия дуги). **Память:** O(V²). **Доступ** к элементу (i, j) и **проверка наличия дуги** — **O(1)**. Перебор всех исходящих из вершины i дуг — O(V). По учебнику 13.1 такое представление **оправдано только для полных или почти полных графов**; при разреженном графе большинство ячеек — нули, использование памяти нерационально.

**Разреженная реализация (словарь).** Вместо хранения всей матрицы хранят только **ненулевые элементы**: ключ — **пара вершин (i, j)** (упорядоченная для ориентированного графа), значение — вес дуги. По учебнику 13.1 целесообразно использовать **реализацию словаря**, лучше всего **хеш-таблицу**, в которой ключом служит пара вершин. **Память:** O(E). **Доступ** к (i, j) — O(1) в среднем для хеша или O(log k) для дерева поиска (k — число хранимых пар). Экономия при **E ≪ V²**.

**Альтернатива: список списков.** Для каждой вершины i хранят список пар (j, weight) только для тех j, для которых есть дуга i→j. Это уже не «матрица» в смысле двумерного массива, а компактное представление той же информации; асимптотика как у списка смежности — O(V+E) память, перебор исходящих из i — O(deg⁺(i)).

**Сравнение.**

| Реализация | Память | Доступ (i,j) | Перебор исходящих из i |
|------------|--------|--------------|-------------------------|
| Плотная (массив V×V) | O(V²) | O(1) | O(V) |
| Разреженная (словарь) | O(E) | O(1) ср. / O(log E) | O(V) без индекса по строкам |
| Список списков | O(V+E) | O(deg⁺(i)) | O(deg⁺(i)) |

```cpp
// Плотная: двумерный массив
std::vector<std::vector<double>> adj(V, std::vector<double>(V, INF));

// Разреженная: словарь (пара → вес), по учебнику 13.1 — ключ-пара; map — O(log E), хеш по паре — O(1) ср.
std::map<std::pair<int,int>, double> sparse_adj;
sparse_adj[{u, v}] = w;
```

**Итог 5.3.** **Плотная** реализация матрицы смежности — массив V×V: O(1) доступ, O(V²) память; оправдана для полных или почти полных графов (учебник 13.1). **Разреженная** — словарь (ключ (i,j) → вес), лучше хеш-таблица: O(E) память, O(1) ср. доступ; экономия при E ≪ V². Список списков — альтернатива с перебором исходящих за O(deg⁺(i)).

## 5.4–5.7. Обход графа в ширину и в глубину (ориентированный и неориентированный)

По учебнику (раздел 13.2): большинство задач на графах сводится к **обходу** — систематическому посещению вершин и дуг. **Основных стратегий обхода две:** в ширину (BFS) и в глубину (DFS). Разница между ними — **лишь в порядке выбора следующей дуги для посещения**: при обходе в ширину используется **очередь (FIFO)**, при обходе в глубину — **стек (LIFO)**. Учебник обобщает оба варианта единым алгоритмом с компонентом **Resolver**: в коллекцию (очередь или стек) добавляются доступные для посещения дуги; на каждом шаге извлекается следующая дуга по правилу Resolver’а (FIFO или LIFO); если дуга ведёт в непосещённую вершину, она помечается посещённой, а её исходящие дуги (не в уже посещённые) добавляются в Resolver (учебник 13.2).

**Общий алгоритм (по учебнику 13.2).** (1) Начать с стартовой вершины, пометить посещённой. (2) Поместить все исходящие из неё дуги в Resolver. (3) Пока Resolver не пуст: извлечь очередную дугу (по FIFO или LIFO); если дуга ведёт в непосещённую вершину — пометить её, добавить в Resolver все её исходящие дуги, ведущие в непосещённые вершины. **Трудоёмкость:** O(V + E) при представлении графа списком смежности (каждая вершина и каждое ребро обрабатываются O(1) раз). Метки посещённости хранят отдельно (массив или множество).

---

### 5.4. Обход ориентированного графа в ширину (BFS)

**Обход в ширину (по учебнику 13.2):** алгоритм сначала посещает все вершины, **смежные с текущей** (соседние), и только затем переходит к вершинам следующего «уровня» удалённости. Для реализации используется **очередь (Queue, FIFO)**: при посещении вершины «раскрывается» её исходящая окрестность, дуги помещаются в очередь и обрабатываются после дуг, соседних по текущему уровню. **Гарантирует нахождение кратчайшего пути в невзвешенном ориентированном графе** (учебник 13.2). Алгоритм: кладём стартовую вершину в очередь и помечаем посещённой; пока очередь не пуста — извлекаем вершину u, для каждой дуги u→v в непосещённую v помечаем v и кладём v в очередь. O(V+E).

---

### 5.5. Обход ориентированного графа в глубину (DFS)

**Обход в глубину (по учебнику 13.2):** алгоритм старается **уйти как можно дальше** от начальной вершины по одному пути и только, дойдя до тупика, возвращается назад и исследует другие ветви. Для реализации используется **стек (Stack, LIFO)** — дуги из исходящей окрестности помещаются в стек; часто реализуют **рекурсией** (стек вызовов играет роль стека дуг). Применения: **топологическая сортировка** в DAG, **поиск циклов**, проверка достижимости. O(V+E).

---

### 5.6. Обход неориентированного графа в ширину (BFS)

**Тот же алгоритм**, что и для ориентированного графа (очередь, помечаем посещённые вершины, добавляем в очередь только непосещённых соседей). В неориентированном графе каждое ребро (u, v) хранится в списках смежности обеих вершин, поэтому при обходе из u мы переходим в v, а при обходе из v — в u; достижимость и «уровни» считаются по рёбрам. **Гарантирует кратчайший путь в невзвешенном неориентированном графе** (по числу рёбер). O(V+E).

---

### 5.7. Обход неориентированного графа в глубину (DFS)

**Тот же алгоритм**, что и для ориентированного (стек или рекурсия, LIFO). Специфика неориентированного случая: при обходе из вершины u по ребру (u, v) мы попадаем в v; из v рёбер обратно в u мы не добавляем в Resolver (v уже посещена) или не идём по нему повторно — иначе каждое ребро проходилось бы дважды. Применения: **компоненты связности** (запуск DFS от непосещённой вершины даёт одну компоненту), **поиск циклов** (если при обходе встречается уже посещённая вершина, не являющаяся родителем, есть цикл). O(V+E).

---

**Примеры кода всех четырёх алгоритмов.** Граф задан списком смежности: `g[u]` — список пар (куда, вес) для исходящих из u дуг; для неориентированного графа каждое ребро (u,v) хранится в `g[u]` и в `g[v]`.

```cpp
#include <queue>
#include <stack>
#include <vector>

using Graph = std::vector<std::vector<std::pair<int, int>>>;  // (куда, вес)
```

**5.4. Обход ориентированного графа в ширину (BFS).**

```cpp
void bfs_directed(const Graph& g, int start) {
    std::vector<bool> visited(g.size(), false);
    std::queue<int> q;
    q.push(start);
    visited[start] = true;
    while (!q.empty()) {
        int u = q.front(); q.pop();
        for (auto [v, w] : g[u])   // исходящие дуги u → v
            if (!visited[v]) {
                visited[v] = true;
                q.push(v);
            }
    }
}
```

**5.5. Обход ориентированного графа в глубину (DFS).** Рекурсия (стек вызовов = LIFO):

```cpp
void dfs_directed_rec(const Graph& g, int u, std::vector<bool>& visited) {
    visited[u] = true;
    for (auto [v, w] : g[u])
        if (!visited[v]) dfs_directed_rec(g, v, visited);
}
```

Итеративно со стеком:

```cpp
void dfs_directed_iter(const Graph& g, int start) {
    std::vector<bool> visited(g.size(), false);
    std::stack<int> st;
    st.push(start);
    visited[start] = true;
    while (!st.empty()) {
        int u = st.top(); st.pop();
        for (auto [v, w] : g[u])
            if (!visited[v]) {
                visited[v] = true;
                st.push(v);
            }
    }
}
```

**5.6. Обход неориентированного графа в ширину (BFS).** Тот же код: в неориентированном графе `g[u]` содержит всех соседей u (каждое ребро хранится с двух концов).

```cpp
void bfs_undirected(const Graph& g, int start) {
    std::vector<bool> visited(g.size(), false);
    std::queue<int> q;
    q.push(start);
    visited[start] = true;
    while (!q.empty()) {
        int u = q.front(); q.pop();
        for (auto [v, w] : g[u])   // все соседи u (рёбра u–v)
            if (!visited[v]) {
                visited[v] = true;
                q.push(v);
            }
    }
}
```

**5.7. Обход неориентированного графа в глубину (DFS).** Рекурсия:

```cpp
void dfs_undirected_rec(const Graph& g, int u, std::vector<bool>& visited) {
    visited[u] = true;
    for (auto [v, w] : g[u])
        if (!visited[v]) dfs_undirected_rec(g, v, visited);
}
```

Итеративно со стеком:

```cpp
void dfs_undirected_iter(const Graph& g, int start) {
    std::vector<bool> visited(g.size(), false);
    std::stack<int> st;
    st.push(start);
    visited[start] = true;
    while (!st.empty()) {
        int u = st.top(); st.pop();
        for (auto [v, w] : g[u])
            if (!visited[v]) {
                visited[v] = true;
                st.push(v);
            }
    }
}
```

**Итог 5.4–5.7.** По учебнику 13.2 обход в ширину и в глубину — две стратегии с общим алгоритмом; различие только в Resolver: **очередь (FIFO)** для BFS, **стек (LIFO)** для DFS. BFS даёт кратчайшие пути в невзвешенном графе; DFS — топологическую сортировку (DAG), поиск циклов, компоненты связности. Ориентированный и неориентированный граф — один и тот же алгоритм; в неориентированном рёбра хранятся с двух концов, при обходе избегают повторного прохода по тому же ребру. Трудоёмкость O(V+E).

## 5.8. Полиморфный АТД «Очередь с приоритетами»: реализация на основе бинарной кучи

По учебнику (раздел 10.8): **основная польза бинарной кучи — её использование для реализации очереди с приоритетами**, где порядок определяется весом (приоритетом) элементов. **Бинарная куча** имеет инвариант: ключи дочерних узлов либо оба **меньше**, либо оба **больше** ключа корня (определяется при создании кучи) — это позволяет иметь **наименьший или наибольший элемент в вершине** (min-heap или max-heap). Бинарную кучу можно **хранить как линейную структуру (массив)**, где дочерне-родительские отношения вычисляются за константное время по формулам: для узла с индексом i дети — 2i+1 и 2i+2, родитель — (i−1)/2 (учебник 10.8).

**Операции очереди с приоритетами на min-heap.** Храним пары (приоритет, элемент) или только ключи; в вершине кучи — элемент с **минимальным приоритетом**. **Insert (вставка):** элемент помещается в конец массива, затем **«поднимается»** (sift-up) до своего места — сравнение с родителем, при нарушении инварианта обмен; **O(log n)**. **ExtractMin (извлечение минимума):** забираем корень, на его место ставим последний элемент, затем **«опускаем»** (sift-down) вниз до восстановления инварианта; **O(log n)**. **Peek (посмотреть минимум без извлечения):** доступ к корню — **O(1)**.

**Полиморфность:** АТД параметризуется типом элемента T и (при необходимости) компаратором приоритета; куча сравнивает элементы по приоритету.

```cpp
// Min-heap: корень — минимум. Хранение в массиве: дети 2*i+1, 2*i+2, родитель (i-1)/2
#include <vector>
template<typename T>
void sift_up(std::vector<T>& heap, int i) {
    while (i > 0) {
        int p = (i - 1) / 2;
        if (heap[p] <= heap[i]) break;
        std::swap(heap[p], heap[i]);
        i = p;
    }
}
template<typename T>
void sift_down(std::vector<T>& heap, int i, int n) {
    while (2*i + 1 < n) {
        int j = 2*i + 1;
        if (j + 1 < n && heap[j+1] < heap[j]) j++;
        if (heap[i] <= heap[j]) break;
        std::swap(heap[i], heap[j]);
        i = j;
    }
}
// insert: heap.push_back(x); sift_up(heap, heap.size()-1);  // O(log n)
// extractMin: T top = heap[0]; heap[0] = heap.back(); heap.pop_back(); sift_down(heap, 0, heap.size()); return top;  // O(log n)
// peek: return heap[0];  // O(1)
```

**Итог 5.8.** По учебнику 10.8 бинарная куча реализует **очередь с приоритетами**: min-heap хранится массивом, в корне — минимум. Insert и ExtractMin — **O(log n)** (sift-up, sift-down), Peek — **O(1)**. Полиморфный АТД параметризуется типом элемента и приоритетом.

## 5.9. Полиморфный АТД «Очередь с приоритетами»: реализация на основе связанного списка

**Очередь с приоритетами** можно реализовать не только на куче, но и на **связанном списке**. Различие — в том, поддерживается ли порядок по приоритету в списке.

**Отсортированный список (по приоритету).** Элементы хранятся в порядке возрастания приоритета (минимум — в голове). **Insert:** поиск позиции для вставки (линейный обход или бинарный поиск при доступе по индексу) — **O(n)**; вставка в найденную позицию — O(1) при наличии указателя на предыдущий узел. **ExtractMin:** удаление головы списка — **O(1)**. **Peek:** доступ к голове — **O(1)**. Подходит, когда извлечений много, а вставок мало.

**Несортированный список.** Элементы добавляются в конец (или в начало). **Insert:** вставка в конец (или в начало) — **O(1)**. **ExtractMin:** поиск минимума обходом по всему списку — **O(n)**; удаление найденного элемента — O(1) при указателе на предыдущий. **Peek:** то же, что ExtractMin без удаления — O(n). Подходит, когда вставок много, а извлечений мало.

**Сравнение с бинарной кучей (5.8).** Куча даёт O(log n) на insert и extractMin, что при смешанной нагрузке обычно лучше, чем O(n) на одну из операций у списка. Список полезен при очень неравномерной нагрузке или когда реализация должна быть максимально простой.

| Реализация | Insert | ExtractMin | Peek |
|------------|--------|------------|------|
| Бинарная куча (5.8) | O(log n) | O(log n) | O(1) |
| Отсортированный список | O(n) | O(1) | O(1) |
| Несортированный список | O(1) | O(n) | O(n) |

```cpp
// Отсортированный список: минимум в голове. Insert — вставка в нужную позицию O(n), ExtractMin — pop_front O(1)
// Несортированный: Insert — push_back O(1), ExtractMin — обход + удаление мин. O(n)
struct Node { int priority; int value; Node* next; };
// ExtractMin для несортированного: найти узел с min priority и его предшественника, удалить, вернуть значение
```

**Итог 5.9.** Очередь с приоритетами на **связанном списке**: **отсортированный список** — insert O(n), extractMin и peek O(1); **несортированный** — insert O(1), extractMin и peek O(n). Выбор зависит от соотношения числа вставок и извлечений; при смешанной нагрузке предпочтительна куча (5.8).

## 5.10. Поиск кратчайших путей: алгоритм Дейкстры

По учебнику (раздел 13.2): алгоритм Дейкстры — классический представитель класса **жадных алгоритмов**; он вписывается в общую схему обхода с **очередью с приоритетами**, где приоритет определяется **минимальной длиной пути** до вершины. Идея: **последовательное расширение подграфа уже посещённых вершин**; очередная дуга выбирается из числа дуг, исходящих из посещённых вершин в непосещённые — та, которая приводит к **минимальному увеличению** текущего «диаметра» (длины самого длинного кратчайшего пути от старта в посещённом подграфе) (учебник 13.2).

**Стандартное изложение (по учебнику 13.2).** (1) Для каждой вершины хранится **текущая минимальная длина пути** от стартовой вершины: изначально 0 для старта и **бесконечность** для остальных. (2) Создаётся **очередь с приоритетами**, в которую помещается стартовая вершина; **приоритет = минимальная длина пути**. (3) На каждом шаге из очереди **извлекается вершина u с наименьшей текущей длиной пути**; эта вершина считается посещённой. (4) Для каждой соседней вершины v для u выполняется **релаксация**: если путь в v через u короче текущего известного пути в v, то dist[v] обновляется, и v добавляется в очередь с новым приоритетом. (5) Процесс повторяется до опустошения очереди или до нахождения путей до всех нужных вершин (учебник 13.2).

**Условие корректности.** **Важным условием корректной работы алгоритма Дейкстры является неотрицательность весов всех дуг** (учебник 13.2). При отрицательных рёбрах кратчайший путь может не существовать или алгоритм даёт неверный результат.

**Реализация.** Граф — список смежности; очередь с приоритетами (min-heap) по ключу dist. При извлечении из очереди вершина может встречаться несколько раз (устаревшие записи с большим dist); при извлечении **пропускаем вершину, если её dist уже обновлён** (текущее значение в массиве dist меньше извлечённого) — так избегаем лишней работы. **Трудоёмкость:** O((V+E) log V) при реализации кучей: каждая дуга может привести к одной вставке в очередь, извлечений не более O(V).

**Восстановление пути.** По учебнику 13.2 для восстановления путей удобно строить **дерево кратчайших путей**: каждый узел хранит ссылку на родителя и дугу перехода; путь восстанавливается движением от конечной вершины вверх к корню. На практике часто хранят массив **parent[v]** — вершина, из которой последний раз обновили dist[v].

```cpp
// Алгоритм Дейкстры: кратчайшие пути из start, веса неотрицательные
#include <queue>
#include <vector>
using Graph = std::vector<std::vector<std::pair<int, long long>>>;  // (куда, вес)

std::vector<long long> dijkstra(const Graph& g, int start) {
    const long long INF = 1e18;
    int n = (int)g.size();
    std::vector<long long> dist(n, INF);
    dist[start] = 0;
    std::priority_queue<std::pair<long long, int>> q;  // min-heap: (-dist, v)
    q.push({0, start});
    while (!q.empty()) {
        auto [d, u] = q.top(); q.pop();
        d = -d;
        if (d != dist[u]) continue;  // устаревшая запись — пропускаем
        for (auto [v, w] : g[u]) {
            if (dist[u] + w < dist[v]) {
                dist[v] = dist[u] + w;
                q.push({-dist[v], v});
            }
        }
    }
    return dist;
}
```

**Итог 5.10.** По учебнику 13.2 алгоритм Дейкстры — жадный поиск кратчайших путей из одной вершины; **очередь с приоритетами** по длине пути, на каждом шаге извлекаем вершину с минимальным dist и **релаксируем** исходящие дуги. **Веса неотрицательные.** Трудоёмкость O((V+E) log V). Устаревшие записи в очереди при извлечении пропускаем. Восстановление пути — по массиву parent или по дереву кратчайших путей.

## 5.11. Динамическое программирование: нисходящий процесс

По учебнику (раздел 13.3): **динамическое программирование (ДП)** — обобщённая методика решения дискретных оптимизационных задач, многие из которых сводятся к поиску оптимального пути в графе состояний. **Основной принцип ДП — принцип оптимальности Беллмана:** оптимальный путь обладает тем свойством, что какова бы ни была последняя вершина на этом пути, предшествующий ей участок также является оптимальным путём до этой вершины. Это позволяет **разбить задачу на подзадачи** и **избежать повторных вычислений**, сохраняя уже найденные решения для подзадач (**мемоизация**) (учебник 13.3).

**Нисходящий процесс (сверху вниз, top-down).** Решение основной задачи **разбивается на подзадачи**, те — на ещё более мелкие, и так далее, пока не будут достигнуты **тривиальные (базовые) случаи**. Этот подход **естественно реализуется рекурсией** и соответствует **обходу графа состояний в глубину** (учебник 13.3). **Мемоизация:** при первом обращении к подзадаче результат вычисляется и сохраняется в кеше (таблица, словарь); при повторном обращении — возврат из кеша. Так экспоненциальная рекурсия превращается в полиномиальную (каждая подзадача решается один раз).

**Пример: числа Фибоначчи.** Рекуррентность F(n) = F(n−1) + F(n−2); базовые случаи F(0)=0, F(1)=1. Без мемоизации — экспоненциальное число вызовов; с мемоизацией — O(n) вызовов, O(n) память.

```cpp
// Нисходящий ДП: рекурсия + мемоизация (кеш при первом обращении)
#include <vector>
long long fib_topdown(int n, std::vector<long long>& memo) {
    if (n <= 1) return n;
    if (memo[n] != -1) return memo[n];  // уже вычислено — возврат из кеша
    memo[n] = fib_topdown(n - 1, memo) + fib_topdown(n - 2, memo);
    return memo[n];
}
// Вызов: std::vector<long long> memo(n + 1, -1); fib_topdown(n, memo);
```

**Итог 5.11.** По учебнику 13.3 **нисходящий ДП** — решение задачи разбиением на подзадачи сверху вниз; реализуется **рекурсией** и **мемоизацией** частичных результатов (обход графа состояний в глубину). Кеш заполняется при обращении; повторные обращения — возврат из кеша. Трудоёмкость определяется числом различных подзадач.

## 5.12. Динамическое программирование: восходящий процесс

По учебнику (раздел 13.3): при **восходящем процессе (снизу вверх, bottom-up)** сначала решаются **все самые мелкие подзадачи**, затем на их основе — подзадачи более крупного уровня, и так далее, пока не будет решена **исходная задача**. Этот подход **обычно реализуется итеративно** (без рекурсии) и соответствует **обходу графа состояний в ширину** (учебник 13.3).

**Реализация.** Заполняем **таблицу** (массив, матрицу) от **базовых случаев** к целевому. Порядок заполнения выбирается так, чтобы при вычислении значения текущей ячейки все нужные подзадачи уже были решены. Рекуррентное соотношение то же, что и при нисходящем подходе; различие — порядок обхода (снизу вверх вместо рекурсивных вызовов).

**Пример: числа Фибоначчи.** Базовые случаи: dp[0]=0, dp[1]=1; для i ≥ 2: dp[i] = dp[i−1] + dp[i−2]. Заполняем массив по возрастанию индекса. O(n) время, O(n) память; при необходимости можно свести память к O(1), храня только два последних значения.

```cpp
// Восходящий ДП: таблица от базовых случаев к целевым, без рекурсии
#include <vector>
long long fib_bottomup(int n) {
    if (n <= 1) return n;
    std::vector<long long> dp(n + 1);
    dp[0] = 0;
    dp[1] = 1;
    for (int i = 2; i <= n; ++i)
        dp[i] = dp[i - 1] + dp[i - 2];
    return dp[n];
}
// Экономия памяти: хранить только dp[i-1] и dp[i-2], обновлять в цикле — O(1) память
```

**Сравнение с нисходящим (5.11).** Нисходящий — рекурсия + мемоизация; вычисляются только те подзадачи, которые реально достигаются из начального состояния. Восходящий — итеративное заполнение таблицы; часто проще по реализации и предсказуемее по порядку обращений; может вычислить лишние подзадачи, если они не все нужны для ответа.

**Итог 5.12.** По учебнику 13.3 **восходящий ДП** — решение подзадач от самых мелких к целевой; реализуется **итеративно**, таблица заполняется от базовых случаев. Соответствует обходу графа состояний в ширину. Трудоёмкость и память определяются размером таблицы; порядок заполнения должен обеспечивать готовность всех зависимостей.

## 5.13. Раскраска графа. Алгоритм Ершова (DSatur)

По учебнику (раздел 13.4): **задача раскраски графа** — присвоить каждой вершине цвет так, чтобы **никакие две смежные вершины не имели одинакового цвета**, используя при этом **минимально возможное количество цветов**. Поскольку задача о нахождении минимальной раскраски **NP-трудная**, на практике используют **эвристические (приближённые) алгоритмы**. Один из таких подходов — **алгоритм Ершова** (учебник 13.4).

**Алгоритм Ершова (по учебнику 13.4).** (1) Выбрать вершину с **наибольшей степенью** и окрасить её в первый цвет. (2) Из 2-окрестности какой-либо окрашенной вершины выбрать неокрашенную; окрасить её в тот же цвет, если она не инцидентна вершине, уже окрашенной в этот цвет; иначе оставить пока. (3) Повторять, пока есть нераскрашенные вершины в 2-окрестности «укрупнённой» вершины. (4) Если остались неокрашенные — выбрать одну (разумно с наибольшей степенью), ввести новый цвет, окрасить её и повторить шаги. (5) Повторять, пока все вершины не будут окрашены. Алгоритм не гарантирует оптимального результата, но часто даёт приемлемое приближение (учебник 13.4).

**DSatur (степень насыщения).** Вариант выбора вершины для окраски: на каждом шаге выбирают вершину с **максимальной степенью насыщения** — числом **различных цветов** среди соседей (вершина «насыщена», если у соседей много разных цветов); при равенстве — с **максимальной степенью**. Выбранную вершину красят в **минимальный допустимый цвет** (наименьший номер цвета, не использованный у соседей). DSatur часто даёт меньше цветов, чем жадный по степени.

```cpp
// DSatur: степень насыщения = число различных цветов у соседей
// На каждом шаге: выбрать вершину с макс. насыщением (при равенстве — макс. степенью)
// Красить в минимальный цвет, не использованный у соседей
int get_saturation(int u, const Graph& g, const std::vector<int>& color) {
    std::set<int> neighbor_colors;
    for (int v : g[u]) if (color[v] != -1) neighbor_colors.insert(color[v]);
    return (int)neighbor_colors.size();
}
// Основной цикл: пока есть неокрашенные — выбрать u с макс. saturation (и макс. deg), 
// color[u] = min { c : c не встречается у соседей }; повторять.
```

**Итог 5.13.** По учебнику 13.4 задача раскраски графа — соседи разного цвета, минимум цветов; NP-трудная. **Алгоритм Ершова** — эвристика: выбор вершины с наибольшей степенью, окраска и расширение по 2-окрестности, при необходимости новый цвет. **DSatur** — выбор вершины с максимальной степенью насыщения (число цветов у соседей), при равенстве — максимальной степенью; красить в минимальный допустимый цвет.

## 5.14. Поиск остова графа. Алгоритм Прима

**Остовное дерево (MST)** — подграф, являющийся деревом и содержащий все вершины; сумма весов рёбер минимальна. **Алгоритм Прима:** на каждом шаге **остов растёт** одной вершиной; выбирается **минимальное по весу ребро**, соединяющее уже включённую в остов вершину с вершиной, ещё не включённой; это ребро и вторая вершина добавляются в остов. Изначально в остов входит одна произвольная вершина (или заданная).

**Реализация.** Поддерживаем множество вершин остова и для каждой внешней вершины — минимальный вес ребра до остова (или бесконечность). На каждом шаге выбираем внешнюю вершину с минимальным таким весом, добавляем её и соответствующее ребро в остов, обновляем веса для соседей новой вершины. Выбор «внешней вершины с минимальным весом до остова» удобно реализовать **очередью с приоритетами (min-heap)** по этому весу. **Трудоёмкость:** O(E log V) при куче (каждое ребро может вызвать одно обновление приоритета в куче).

```cpp
// Алгоритм Прима: MST, растущее дерево, min ребро (дерево → внешняя вершина)
#include <queue>
#include <vector>
using Graph = std::vector<std::vector<std::pair<int, long long>>>;

long long prim(const Graph& g, int start = 0) {
    int n = (int)g.size();
    std::vector<bool> in_mst(n, false);
    std::vector<long long> min_edge(n, 1e18);
    min_edge[start] = 0;
    std::priority_queue<std::pair<long long, int>> q;  // (-вес, вершина) для min-heap
    q.push({0, start});
    long long total = 0;
    while (!q.empty()) {
        auto [w, u] = q.top(); q.pop();
        w = -w;
        if (in_mst[u]) continue;
        in_mst[u] = true;
        total += w;
        for (auto [v, wuv] : g[u])
            if (!in_mst[v] && wuv < min_edge[v]) {
                min_edge[v] = wuv;
                q.push({-wuv, v});
            }
    }
    return total;
}
```

**Итог 5.14.** Алгоритм Прима строит **остовное дерево минимального веса**: на каждом шаге добавляется **минимальное ребро** из текущего остова во внешнюю вершину. Реализация с очередью с приоритетами по весу ребра до остова. Трудоёмкость O(E log V).

## 5.15. Поиск остова графа. Алгоритм Краскала

**Алгоритм Краскала:** рёбра рассматриваются в порядке **неубывания веса**; ребро добавляется в остов, если оно **не создаёт цикла** с уже выбранными рёбрами (т.е. соединяет две разные компоненты связности текущего подграфа). Изначально каждая вершина — отдельная компонента; в конце получается одно дерево — MST.

**Реализация.** Сортируем все рёбра по весу — O(E log E). Для проверки «создаёт ли ребро (u, v) цикл» достаточно проверить, лежат ли u и v в **одной компоненте связности**. Структура **Union-Find (DSU)** поддерживает операции: **find(v)** — представитель компоненты v; **unite(u, v)** — объединить компоненты u и v. Реализация с **сжатием пути** в find и **объединением по рангу** (или по размеру) в unite даёт практически O(1) на операцию в среднем. Итого: O(E log E) на сортировку + O(E · α(V)) на проверки — **O(E log E)**.

```cpp
// Алгоритм Краскала: рёбра по возрастанию веса, добавляем если не цикл (Union-Find)
#include <algorithm>
#include <vector>

struct Edge { int u, v; long long w; };
struct DSU {
    std::vector<int> parent, rank;
    DSU(int n) : parent(n), rank(n, 0) { for (int i = 0; i < n; ++i) parent[i] = i; }
    int find(int x) {
        if (parent[x] != x) parent[x] = find(parent[x]);  // сжатие пути
        return parent[x];
    }
    bool unite(int x, int y) {
        x = find(x); y = find(y);
        if (x == y) return false;
        if (rank[x] < rank[y]) std::swap(x, y);
        parent[y] = x;
        if (rank[x] == rank[y]) ++rank[x];  // по рангу
        return true;
    }
};

long long kruskal(int n, std::vector<Edge>& edges) {
    std::sort(edges.begin(), edges.end(), [](const Edge& a, const Edge& b) { return a.w < b.w; });
    DSU dsu(n);
    long long total = 0;
    for (auto& e : edges)
        if (dsu.unite(e.u, e.v)) total += e.w;
    return total;
}
```

**Итог 5.15.** Алгоритм Краскала строит **MST**: рёбра сортируются по весу, добавляем ребро, если оно **не соединяет вершины одной компоненты** (проверка через **Union-Find**). Find со сжатием пути, unite по рангу — практически O(1) на операцию. Трудоёмкость O(E log E).
