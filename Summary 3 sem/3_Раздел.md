## 3.1. Концепция ленивой последовательности. Логическое определение и внешняя спецификация

**Что такое ленивая последовательность.** Ленивая последовательность — это последовательность элементов, у которой **хвост (tail) и последующие элементы не вычисляются при создании**, а вычисляются **только при обращении** (при вызове tail(), при переходе к следующему элементу). То есть «ленивость» — отложенное вычисление: следующий элемент (и вся оставшаяся последовательность) создаётся не в момент построения текущего узла, а в момент, когда к нему обращаются. Это позволяет представлять **бесконечные** последовательности (например, натуральные числа 0, 1, 2, …) и последовательности, порождённые генератором, без предварительного вычисления всех элементов в памяти.

**Ленивое vs строгое вычисление.** В **строгой** (eager) последовательности при создании Cons(head, tail) значение tail уже вычислено (например, указатель на следующий узел списка). В **ленивой** последовательности tail — это не готовое значение, а **обещание** (thunk) или **генератор**: при первом обращении к tail() выполняется вычисление, результат запоминается (мемоизация), при следующих обращениях возвращается уже вычисленное значение. Поэтому создание Cons(0, naturals(1)) не запускает построение всего списка натуральных чисел — только при вызове tail() начнётся вычисление следующего элемента.

**Логическое определение.** Ленивая последовательность задаётся **рекурсивно**:

- **Nil** — пустая последовательность (конец списка).
- **Cons(head, tail)** — непустая последовательность: **head** — первый элемент (уже известное значение), **tail** — остаток последовательности, вычисляемый **лениво** (при обращении).

То есть структура та же, что у обычного связного списка (голова + указатель на хвост), но хвост не хранится как готовый узел, а как **отложенное вычисление** (функция без аргументов, при вызове возвращающая следующую последовательность: Nil или Cons). Рекурсивно: tail сам есть Nil или Cons(next_head, next_tail), где next_tail снова ленивый. Таким образом, последовательность может быть бесконечной: например, naturals = Cons(0, lazy(naturals(1))), где naturals(1) = Cons(1, lazy(naturals(2))), и т.д.

**Внешняя спецификация (интерфейс LazySequence).** Ниже приведена спецификация в духе таблицы 9.1: создание объекта, декомпозиция (доступ к элементам и подпоследовательностям), операции. Элементы и хвосты **вычисляются при обращении** (с мемоизацией), а не при создании.

**Создание объекта.**

| Конструктор | Назначение |
|-------------|------------|
| `LazySequence()` | Пустая последовательность. |
| `LazySequence(T* items, int count)` | Копировать элементы из переданного массива. |
| `LazySequence(Sequence<T>* seq)` | Копировать элементы из переданной последовательности. |
| `LazySequence(T (*)(Sequence<T>*), Sequence<T>)` или `LazySequence(std::function<T(Sequence<T>*)>, Sequence<T>*)` | Задание **(рекуррентного) порождающего правила**: функция использует k предшествующих элементов для порождения (n+1)-го, где k ≤ n, а также задаётся список первых k элементов. |

| `T GetLast()` | IndexOutOfRange (если пуста). Вычисление может не завершиться при бесконечной последовательности. | Получить последний элемент. |
| `T Get(size_t index)` | IndexOutOfRange (если индекс отрицательный или ≥ числа элементов). Может не завершиться при бесконечной последовательности. | Получить элемент по индексу (упрощённая версия: индекс — неотрицательное целое). |
| `LazySequence<T>* GetSubsequence(int startIndex, int endIndex)` | IndexOutOfRange (если индекс отрицательный или ≥ числа элементов). | Подпоследовательность от startIndex до endIndex. |
| `Cardinal GetLength() const` | — | Длина последовательности. Тип **Cardinal** — множество обычных натуральных чисел и **первое трансфинитное число** (для бесконечных последовательностей). |
| `size_t GetMaterializedCount() const` | — | Количество **материализованных** (уже вычисленных) элементов. **Служебная** характеристика, не входит в логическое определение АТД; с точки зрения абстракции такой операции быть не должно — она раскрывает детали реализации. |

**Операции.**

| Метод | Исключения / особенности | Назначение |
|-------|--------------------------|------------|

| `Sequence<T>* Prepend(T item)` | — | Добавить элемент в начало. |
| `LazySequence<T>* InsertAt(T item, int index)` и перегрузки с `Sequence<T>*` / `LazySequence<T>*` | IndexOutOfRange (если индекс отрицательный или ≥ числа элементов). | Вставить элемент или (возможно, бесконечную) последовательность в заданную позицию. |
| `LazySequence<T>* Skip(int index)` / `Skip(size_t start, size_t end)` | IndexOutOfRange. | «Удалить» (пропустить) подпоследовательность. **Фактического удаления нет** — строится новая ленивая последовательность, в которой при обходе эти элементы не выдаются. |
| `LazySequence<T>* Replace(...)` (один индекс или диапазон + последовательность) | IndexOutOfRange. | Заменить элемент или подпоследовательность на другую последовательность. |
| `LazySequence<T>* Concat(LazySequence<T>* list)` | IndexOutOfRange. | Сцепить две последовательности. Если первая **бесконечная**, до элементов второй при перечислении (например, итератором) можно «дойти» только в бесконечности. |
| `LazySequence<T2>* Map(...)`, `T2 Reduce(...)`, `LazySequence<T>* Where(...)`, `LazySequence<...>* Zip(...)` и т.д. | IndexOutOfRange при выходе за границы. | Операции группы **map-reduce** и родственные (Where, Zip). Реализация принципиально не отличается от обычных последовательностей: используется либо доступ по индексу, либо перечисление (итерация), что и позволяет абстрагироваться от ленивости и мемоизации. |

**Соответствие с логическим определением.** GetFirst соответствует **head**, получение хвоста (остатка после первого элемента) — через GetSubsequence(1, …) или ленивое вычисление следующей последовательности (аналог **tail**). Get(index) для индекса k по смыслу реализуется как «k раз перейти к хвосту, затем GetFirst» (см. билет 3.3). Пустота проверяется через длину или отдельный предикат (в минимальной спецификации — isEmpty / проверка Nil).

**Важно:** элементы (в частности, хвосты и результаты GetSubsequence) **вычисляются при обращении**, а не при создании. Создание последовательности с порождающим правилом не вычисляет все элементы; при первом обращении к очередному фрагменту выполняется вычисление, результат при необходимости запоминается (мемоизация). Так достигается ленивость: память и время тратятся только на уже запрошенную часть последовательности.

## 3.2. Концепция ленивой последовательности. Основные приемы реализации

По учебнику (раздел 9.1): **ленивая последовательность (LazySequence)** — ситуация, когда есть **порождающее правило** для вычисления i-го элемента (возможно, по ранее вычисленным), последовательность может быть **бесконечной**. Примеры: факториалы {n!}, Фибоначчи. Ленивый список — **линейная структура с последовательным доступом**; вычисление i-го элемента в общем случае требует всех предшествующих, но при **мемоизации** уже вычисленных значений **среднее время** доступа к первым n элементам стремится к **O(1)**. Последовательность может быть **конечной** (файл, сеть — материализовать целиком нежелательно) или **бесконечной** (учебник 9.1).

**Три приёма реализации.** **(1) Не материализовать результат операций:** при вставке/удалении не строить новый список — хранить **ссылку на исходную** последовательность и **очередь (список) изменений** (позиция + тип изменения). **(2) Генератор:** объект с методом **GetNext()**, инкапсулирующий правило порождения следующего элемента (рекуррентное или нет); один на список, внутренний, может менять состояние при мемоизации. **(3) Мемоизация:** при первом обращении к элементу — вычисление (через GetNext() или по правилу) и **запоминание**; при повторном — возврат из кеша. Поэтому у ленивой последовательности практически **нет константных методов**. Эквивалентные идеи в других источниках: **thunk** (отложенное вычисление), **замыкание-генератор**; в C++20 — **coroutines** (`generator<T>`, `yield`).

**Пример: генератор и мемоизация (ленивая последовательность натуральных чисел).**

```cpp
// Генератор — правило порождения следующего элемента; GetNext() возвращает очередной
class NatGenerator {
    int current = 0;
public:
    int GetNext() { return current++; }
};

// Ленивая последовательность с мемоизацией: первый запрос к [i] вычисляет и кеширует
class LazyNats {
    std::vector<int> memo;  // уже вычисленные элементы
    std::unique_ptr<NatGenerator> gen;
public:
    LazyNats() : gen(std::make_unique<NatGenerator>()) {}
    int operator[](size_t i) {
        while (memo.size() <= i)
            memo.push_back(gen->GetNext());  // материализуем по требованию
        return memo[i];
    }
};
// LazyNats seq; int a = seq[0], b = seq[1];  // 0, 1; seq[0] второй раз — из memo
```

**Пример: замыкание-генератор (Fibonacci) — идея thunk/ленивого вычисления.**

```cpp
// Функция с захваченным состоянием возвращает следующий элемент (идея генератора)
auto make_fib_generator() {
    long long a = 0, b = 1;
    return [=]() mutable {
        long long next = a;
        a = b;
        b = next + b;
        return next;
    };
}
// auto next = make_fib_generator(); next(); next(); next();  // 0, 1, 1, 2, ...
```

**Итог 3.2.** По учебнику 9.1 ленивая последовательность — порождающее правило, конечная или бесконечная; мемоизация даёт O(1) в среднем для доступа к префиксу. Приёмы: **(1)** не материализовать — ссылка на исходную + очередь изменений; **(2)** генератор GetNext(); **(3)** мемоизация при первом обращении, нет константных методов.

## 3.3. Понятие ординального числа. Использование ординалов для индексирования последовательностей

**Формальное введение ординалов.** Ординалы (ординальные числа) вводятся как **характеристика множеств**: число, связываемое с **порядковым типом** того или иного **линейно упорядоченного** множества. Пусть M — линейный порядок. **Порядковым типом** M называется совокупность всех линейных порядков, **изоморфных** M. Эта совокупность образует класс эквивалентности, поэтому достаточно рассматривать одного представителя; удобно использовать числовые последовательности: вместо произвольных последовательностей из n элементов {a₁,…,aₙ}, {b₁,…,bₙ} можно рассматривать {0, 1, …, n−1}. **Начальным отрезком** последовательности {a₁, …, aₖ₋₁, aₖ, …}, **отсекаемым** элементом aₖ, называется последовательность {a₁, …, aₖ₋₁}. В терминах порядка: начальный отрезок линейного порядка ⟨A, ≤⟩, отсекаемый элементом a ∈ A, — это {x | x ∈ A ∧ x ≤ a}. Элементами «последовательности порядковых типов» служат такие начальные отрезки ряда натуральных чисел (и их обобщения).

**Обозначение ординалов — начальные отрезки.** Начальные отрезки удобно обозначать числом, следующим **непосредственно за последним** элементом отрезка:

| Начальный отрезок | Ординал |
|-------------------|--------|
| ∅ | 0 |
| {0} | 1 |
| {0, 1} | 2 |
| … | … |
| {0, 1, …, n−1} | n |

Для конечных последовательностей это совпадает с количеством элементов (мощностью). Вопросы возникают, когда переходим к бесконечным последовательностям.

**Первое трансфинитное ординальное число ω.** Первая бесконечная последовательность — весь ряд натуральных чисел **N** = {0, 1, 2, …}. Его мощность — первое трансфинитное кардинальное число ℵ₀; для **порядкового типа** в этом контексте принято обозначение **ω**. Таким образом:

| Начальный отрезок | Ординал |
|-------------------|--------|
| … | … |
| {0, 1, …, n−1} | n |
| … | … |
| {0, 1, 2, …} = N | **ω** |

**Ординалы за ω.** Существуют порядковые типы «длиннее» ω. Пример: возьмём чётные натуральные {0, 2, 4, …} — порядковый тип снова ω (равномощно N). Добавим **в конец** один элемент: {0, 2, 4, …, 1} — это уже другой порядковый тип, его обозначают **ω + 1**. Продолжая: {0, 2, 4, …, 1, 2} ↔ **ω + 2**, …, {0, 2, 4, …, 1, 2, 3, …, n} ↔ **ω + n**. Неформально: после того как «закончили» перечислять бесконечный набор, продолжаем перечисление следующими элементами. **Сцепление двух** бесконечных последовательностей даёт порядковый тип **ω + ω = ω·2** (не 2ω, т.к. 2ω = ω). Сцепление ω раз по ω элементов: **ω + ω + … + ω** (ω раз) = **ω²**. Процесс можно продолжать: ω³, ω⁴, …, ω^ω, … Для наших целей достаточно конечных ординалов, ω и при необходимости ω·2, ω².

**Арифметика ординалов.** Ординалы образуют алгебру с **некоммутативными** сложением и умножением: **2 + ω = ω ≠ ω + 2** и **2·ω = ω ≠ ω·2**. Порядок слагаемых/множителей существен.

**Ординалы и индексирование последовательностей.** В контексте последовательностей (в т.ч. ленивых) **ординал задаёт позицию** (порядковый тип начального отрезка до этой позиции). Конечные ординалы 0, 1, 2, …, n: позиция 0 — «перед первым элементом» (начальный отрезок ∅), позиция k — после k-го элемента (начальный отрезок из k элементов). Ординал **n** для последовательности из n элементов — позиция **«за последним»** (past-the-end), аналог **end()** в C++. Для **бесконечной** последовательности (порядковый тип ω) позиции — 0, 1, 2, …; «позиция после всех» — ординал **ω** (до неё при перечислении «не дойти»). В спецификации LazySequence длина возвращается как тип **Cardinal** (натуральные + первое трансфинитное число), что позволяет выражать длину бесконечной последовательности как ω.

**Доступ по ординалу в ленивой последовательности.** У ленивой последовательности нет произвольного доступа за O(1): есть только **head** и **tail**. Элемент в позиции (конечного ординала) **k** получается как **k раз tail, затем head**: s₀ = s; для i = 0, …, k−1 выполнить s_{i+1} = tail(s_i); вернуть head(s_k). Стоимость — O(k). Если элементов меньше k (встретили Nil), обращение по ординалу k не определено (IndexOutOfRange). Позиция past-the-end (ординал n для n элементов) не содержит элемента — разыменовывать нельзя (как end() в C++). **Связь с итераторами:** begin() ↔ ординал 0, end() ↔ past-the-end (ординал n или ω), ++it ↔ одно применение tail, \*it ↔ head.

**Представление ординалов в программе.** Для ординалов не выше **ω²** достаточно упорядоченных пар: **(p, k)** представляет ординал **ω·p + k**. В частности, ординалы меньше ω — пары (0, k). Примерная идея класса (C++): `Ordinal::Zero()`, `Ordinal::Number(size_t)` (= ω·0 + number), `Ordinal::Omega(size_t)` (= ω·omega + 0), `Ordinal::From(omega, number)`, `Ordinal::OmegaSquare()`; операции +, −, ==, <, ≤. Для больших ординалов ординалы представляют по аналогии с многочленами — коэффициенты при степенях ω (список по возрастанию степени), например ω³ + ω·2 + 5 как [5, 0, 2, 1]; для ещё больших — степень тоже может быть ординалом (например, ω^(ω³)·4).

**Итог.** Ординал — порядковый тип линейно упорядоченного множества (класс изоморфных порядков); обозначают начальные отрезки числами 0, 1, …, n, ω, ω+1, …, ω·2, ω², … Арифметика ординалов некоммутативна. Для последовательностей ординал задаёт позицию; past-the-end — ординал n (или ω для бесконечной). В ленивой последовательности доступ по конечному ординалу k — «k раз tail, затем head», стоимость O(k). В реализации ординалы до ω² можно представлять парой (p, k) = ω·p + k.

## 3.4. Арифметика ленивых последовательностей. Конкатенация, вставка. Удаление подпоследовательности

Под **арифметикой** ленивых последовательностей понимают операции, которые строят новые последовательности из существующих, **не вычисляя лишних элементов заранее**: конкатенация (склейка двух последовательностей), вставка элемента в заданную позицию и удаление подпоследовательности. Все операции формулируются через **head**, **tail** и рекурсию и остаются ленивыми: элементы второй последовательности (или хвоста после вставки/удаления) вычисляются только при обращении.

---

**Конкатенация (concat).** Результат конкатенации последовательностей **A** и **B** — последовательность, в которой сначала идут все элементы A, затем все элементы B. Ленивое определение:

- Если **A пуста (Nil)** — результат конкатенации есть **B** (лениво: B не вычисляется, пока к ней не обратятся).
- Если **A непуста**, A = Cons(h, t), то результат — **Cons(h, lazy(concat(t, B)))**. То есть голова результата — голова A; хвост результата — **ленивая** конкатенация хвоста A и B. При обращении к tail результата будет вычисляться concat(tail(A), B), и так далее, пока не «израсходуется» A; затем начнут выдаваться элементы B.

Псевдокод:
```
concat(A, B):
  if isEmpty(A): return B
  return Cons( head(A), lazy(() => concat(tail(A), B)) )
```
Таким образом, элементы B не требуются и не вычисляются до тех пор, пока не будет пройдена вся последовательность A. Конкатенация **ленива** по второй последовательности и по хвосту первой.

---

**Вставка элемента в позицию k.** Вставить элемент **x** в позицию k означает: взять префикс последовательности до позиции k (элементы с ординалами 0, …, k−1), затем один раз выдать x, затем хвост исходной последовательности (элементы с ординалами k, k+1, …). То есть результат = **concat(prefix(k), Cons(x, suffix))**, где prefix(k) — последовательность первых k элементов, suffix — последовательность после позиции k (получается применением tail k раз от текущей позиции).

Алгоритм (лениво):

1. Если **k = 0**: результат = **Cons(x, s)** — вставка в начало.
2. Если **k > 0** и последовательность пуста — ошибка (позиции нет).
3. Если **k > 0** и последовательность непуста: голова результата — head(s), хвост результата — **ленивая** вставка x в позицию k−1 в tail(s). То есть `insertAt(s, k, x) = Cons(head(s), lazy(() => insertAt(tail(s), k-1, x)))`.

Псевдокод:
```
insertAt(s, k, x):
  if k == 0: return Cons(x, s)
  if isEmpty(s): error "позиция за пределами"
  return Cons( head(s), lazy(() => insertAt(tail(s), k-1, x)) )
```
Элементы исходной последовательности после позиции k вычисляются только при обращении к соответствующей части результата. Вставка остаётся ленивой.

---

**Удаление подпоследовательности [k, k+m).** Удалить отрезок с позиции **k** длины **m** означает: оставить элементы с ординалами 0, …, k−1, затем пропустить m элементов, затем приклеить всё, что идёт после позиции k+m. Результат = **concat(prefix(k), skip(s, m))** после того, как мы «дошли» до позиции k в s; здесь skip(s, m) — последовательность после пропуска m элементов от s.

Алгоритм (лениво):

1. Если **k = 0**: нужно пропустить m элементов от начала, затем вернуть оставшийся хвост. То есть **drop(s, m)** — применить tail m раз (лениво), результат — новая последовательность.
2. Если **k > 0**: голова результата — head(s), хвост — ленивое удаление [k−1, k−1+m] из tail(s): `removeRange(s, k, m) = Cons(head(s), lazy(() => removeRange(tail(s), k-1, m)))`.
3. Функция **drop(s, m)**: если m = 0, вернуть s; иначе вернуть ленивый drop(tail(s), m−1).

Псевдокод:
```
drop(s, m):   // пропустить m элементов от начала
  if m == 0: return s
  if isEmpty(s): return Nil  // или ошибка
  return lazy(() => drop(tail(s), m-1))

removeRange(s, k, m):  // удалить подпоследовательность [k, k+m)
  if k == 0: return drop(s, m)
  if isEmpty(s): error "позиция за пределами"
  return Cons( head(s), lazy(() => removeRange(tail(s), k-1, m)) )
```
Элементы после удаляемого участка не вычисляются, пока до них не дойдут при обходе результата. Удаление — ленивая операция.

---

**Итог.** Конкатенация, вставка и удаление подпоследовательности на ленивых последовательностях задаются рекурсивно через head/tail и отложенный вызов (lazy). Элементы «второй части» или хвоста вычисляются только при обращении, что сохраняет ленивость и позволяет работать с длинными и даже бесконечными последовательностями без предварительного вычисления всего списка.

## 3.5. «Двунаправленная» ленивая последовательность. Приемы реализации

По учебнику (раздел 9.4): в потоке только для чтения предусмотрена возможность **вернуться в ранее прочитанную позицию** — метод **IsCanGoBack()**. Для источников без отката (сокет, генератор) это возможно только при **запоминании** уже прочитанных элементов; операция почти всегда имеет линейную сложность (учебник 9.4).

**Двунаправленная последовательность** — можно двигаться **вперёд** (следующий элемент из генератора/источника) и **назад** (к уже пройденным). Источник выдаёт только вперёд, поэтому «назад» реализуется **буфером пройденных элементов**: вперёд — читаем из источника и **добавляем в буфер**, назад — **читаем из буфера** без обращения к источнику. Буфер — **стек** (push при вперёд, pop при назад) или **список с курсором** (индекс в уже прочитанных). **Кольцевой буфер** размера N — храним только последние N элементов; назад не более чем на N шагов, память O(N).

**Пример: двунаправленная обёртка над генератором с буфером и курсором.**

```cpp
// Буфер пройденных элементов + курсор; вперёд — из генератора в буфер, назад — по буферу
template<typename T>
class BidirLazySeq {
    std::function<T()> gen_;      // генератор (GetNext)
    std::vector<T> buffer_;       // уже прочитанные элементы
    size_t pos_ = 0;              // текущая позиция (индекс в buffer_)
public:
    explicit BidirLazySeq(std::function<T()> gen) : gen_(std::move(gen)) {}

    bool hasNext() const { return true; }  // упрощённо: генератор бесконечный
    T next() {
        if (pos_ < buffer_.size()) return buffer_[pos_++];  // уже есть — из буфера
        T x = gen_();
        buffer_.push_back(x);
        pos_ = buffer_.size();
        return x;
    }
    bool canGoBack() const { return pos_ > 0; }
    T prev() {
        if (pos_ == 0) throw std::runtime_error("no prev");
        --pos_;
        return buffer_[pos_];  // сдвиг назад, вернуть последний выданный при next()
    }
};
// Использование: next() — вперёд (из gen + в buffer), prev() — назад (из buffer)
```

**Итог 3.5.** По учебнику 9.4 возврат назад — IsCanGoBack(); без отката у источника нужна **буфер пройденных**. Двунаправленная ленивая последовательность: вперёд — генератор + добавление в буфер, назад — чтение из буфера. Для ограничения памяти — кольцевой буфер (назад не более N шагов).

## 3.6. Поток данных (stream). Назначение и принцип реализации

По учебнику (раздел 9.4): структуры с последовательным доступом иногда рассматривают через призму т.н. **потоков** (stream), в которых чтение, как правило, рассматривается **логически или физически как удаление элемента из последовательности** (хотя бывают и исключения). Потоки во многом **похожи на ленивые последовательности**, однако отличаются тем, что **для потоков не определена алгебра** — нет таких операций, как конкатенация. Как у всякой линейной структуры, у потока есть **начало** (начальный или первый элемент) и **конец** (последний или конечный элемент). Потоки как правило **конечны**, но нередко ведут себя как (конечные) ленивые последовательности (например, получение данных через сокеты нередко рассматривается как чтение из потока). **Назначение** потока — абстрагировать последовательный доступ к данным из разных источников через единый интерфейс; код обработки не зависит от того, откуда приходят данные.

**Виды потоков.** Потоки бывают: **только для чтения** (получение начального элемента, который сразу же «удаляется» из потока); **только для записи** (добавление элемента в конец); **смешанный вариант**. Если в ленивых списках элементы задаются **процедурно**, то в потоках они **берутся из тех или иных источников** — последовательностей различного типа, хранящихся в памяти, из файлов, из других потоков (например, потоков ввода-вывода).

---

**Внешняя спецификация потока только для чтения (ReadOnlyStream)** — по таблице 9.2 учебника.

**Создание объекта:** ReadOnlyStream(Sequence<T>*), ReadOnlyStream(LazySequence<T>*), ReadOnlyStream(file, Deserializer<T>), ReadOnlyStream(string, Deserializer<T>), ReadOnlyStream(Stream). Различные варианты создать поток: из последовательности элементов, из файла или строки (с указанием способа десериализации объектов типа T из строки), из другого потока.

**Декомпозиция:**

| Метод | Исключения / назначение |
|-------|-------------------------|
| **bool IsEndOfStream() const** | Возвращает признак того, что достигнут конец потока и дальнейшее чтение невозможно. |
| **T Read()** | EndOfStream (если достигнут конец потока). Считывает текущий элемент потока и переходит на следующую позицию. |
| **size_t GetPosition() const** | Возвращает текущую позицию в потоке (по сути — сколько элементов было считано на данный момент). |
| **bool IsCanSeek() const** | Возвращает, возможно ли перемещение вдоль по потоку **без чтения**. Возможно, например, если данные в sequence или в файле; невозможно — например, если данные читаются из сокета. |
| **size_t Seek(size_t index)** | EndOfStream при недостижимой позиции. Пробует перейти на заданную позицию. Можно рассматривать как аналог операции Skip на списках. |
| **bool IsCanGoBack() const** | Возвращает, возможно ли **вернуться в ранее прочитанную позицию**. При чтении файлов в зависимости от режима открытия это бывает возможным; операция почти всегда имеет линейную сложность. |

**Операции:** **void Open()** — после создания объекта могут требоваться дополнительные действия, прежде чем станет возможным чтение данных (инкапсулируются в Open). **void Close()** — когда работа с потоком завершена, финализирующие действия (например, освобождение дескрипторов файлов).

---

**Поток только для записи** — аналогично задаётся внешняя спецификация (таблица 9.3): создание из последовательности, файла, строки, другого потока; операции записи; при необходимости Open/Close. На практике встречаются и потоки, **совмещающие чтение и запись** (в первую очередь при работе с файлами); их возможности — комбинация потоков только для чтения и только для записи.

**Принцип реализации.** Единый интерфейс (IsEndOfStream, Read, GetPosition, при необходимости Seek, IsCanGoBack, Open, Close) и **конкретные классы** для каждого типа источника: последовательность в памяти, файл (с десериализатором), строка, другой поток. Для файла и сети часто используют **буферизацию** (чтение блоками), чтобы уменьшить число обращений к диску или сетевые задержки. Связь с ленивыми последовательностями: поток похож на последовательность с последовательным доступом и явным состоянием (позиция); в отличие от ленивых последовательностей, у потока нет алгебры (конкатенация и т.п.) и чтение обычно «потребляет» элемент (логически или физически удаляет его из потока).

**Итог.** Поток данных (stream) — абстракция последовательного доступа; чтение обычно трактуется как удаление элемента из последовательности; алгебра для потоков не определена. **Назначение** — единый интерфейс для чтения (и записи) из разных источников (память, файл, строка, другой поток). **Принцип реализации** — внешняя спецификация ReadOnlyStream (и потока для записи) по таблицам 9.2–9.3; конкретные классы для каждого источника; при необходимости Open/Close, буферизация, Seek и IsCanGoBack (для возврата в ранее прочитанную позицию).

## 3.7. Поток данных. Пример реализации потока, считывающего данные из бинарного файла

```cpp
class FileByteStream {
    FILE* f_;
    uint8_t buf_[4096];
    size_t bufPos_, bufLen_;
    bool fillBuf() {
        bufLen_ = fread(buf_, 1, sizeof(buf_), f_);
        bufPos_ = 0;
        return bufLen_ > 0;
    }
public:
    bool hasNext() { return bufPos_ < bufLen_ || !feof(f_); }
    uint8_t read() {
        if (bufPos_ >= bufLen_) fillBuf();
        return buf_[bufPos_++];
    }
};
```
Буферизация уменьшает обращения к диску.

template<typename T>
class ArrayStream {

public:

## 3.9. Кодирование потока данных, пример алгоритма

**Что такое кодирование потока данных.** **Кодирование потока** — это преобразование данных при **последовательном чтении** из входного потока и **последовательной записи** во выходной поток. Алгоритм не загружает все данные в память: читает по элементу (или блоку), обрабатывает и пишет результат. Так можно сжимать, кодировать или фильтровать данные произвольного размера (файлы, сетевой трафик), используя только интерфейс потоков (hasNext, read, write). **Пример алгоритма** — RLE (кодирование длин серий): замена повторяющихся подряд одинаковых символов на пару (символ, длина серии).

---

**RLE (Run-Length Encoding).** Идея: если во входных данных подряд идут одинаковые символы (например, «aaaaabbb»), хранить их как **серии** — пары «символ + количество повторений». Вместо «aaaaabbb» записываем, например, (a, 5) и (b, 3). Так достигается **сжатие**, когда повторений много; если повторений мало или данные разнообразны, RLE может **увеличить** размер (одна пара может занимать больше места, чем один символ). RLE эффективен для изображений с длинными горизонтальными полосами одного цвета, простых растров, логов с повторяющимися символами.

**Формат закодированных данных.** Варианты представления пары (символ c, длина n):

- **Литерально:** сначала байт c, затем байт (или слово) n. Декодер читает пары (c, n) и выводит c ровно n раз.
- **Специальный escape:** если n = 1, можно выводить просто c (чтобы не раздувать данные без повторений); если n > 1 — специальный маркер + c + n. Тогда декодер различает «одиночный символ» и «серию».
- В простейшем варианте **всегда** пара (c, n): даже для одиночного символа записываем (c, 1). Декодирование единообразно.

Ниже рассмотрен вариант «всегда пара»: кодируем каждую серию как (prev, count), декодируем — читаем пары и выводим символ count раз.

---

**Кодирование (сжатие).** Алгоритм в один проход по входному потоку:

1. Прочитать первый символ в **prev**, счётчик **count = 1**.
2. Пока во входном потоке есть данные: прочитать следующий символ **c**. Если **c == prev** — увеличить count (серия продолжается). Иначе — **записать** текущую серию: write(prev, count); затем начать новую серию: prev = c, count = 1.
3. После окончания потока записать последнюю серию: write(prev, count).

Таким образом, мы «сворачиваем» подряд идущие одинаковые символы в пары (символ, длина) и пишем их во выходной поток. Память — O(1); один проход.

```cpp
// Вход: поток in (hasNext(), read()); выход: поток out (write(symbol, count))
char prev = in.read();
int count = 1;
while (in.hasNext()) {
    char c = in.read();
    if (c == prev) ++count;
    else {
        out.write(prev, count);  // записать пару (prev, count)
        prev = c;
        count = 1;
    }
}
out.write(prev, count);
```

---

**Декодирование (разжатие).** Алгоритм: читать из закодированного потока **пары** (символ c, число n) и для каждой пары **выводить символ c ровно n раз**. Один проход по закодированным данным; память O(1).

Псевдокод:
```
пока есть следующая пара (c, n):
    для i = 1 .. n: вывести c
```

Если пары хранятся как подряд два байта (сначала символ, затем длина — байт или слово), декодер читает два значения и выводит символ n раз. Пример на C++ (если write при кодировании записывал пару как два байта: символ и счётчик):

```cpp
while (in.hasNext()) {
    char c = in.read();
    int n = in.read();  // или readCount() — длина как байт/слово
    for (int i = 0; i < n; ++i) out.write(c);
}
```

(В реальности тип для n может быть больше байта, если серии длинные; формат зависит от соглашения.)

---

**Связь с потоками.** Кодирование и декодирование работают **строго через потоки**: чтение из входного потока (read, hasNext) и запись во выходной (write). Не требуется загружать весь файл в память — данные обрабатываются последовательно. Это типичный пример алгоритма кодирования потока данных.

**Итог.** Кодирование потока данных — преобразование при последовательном чтении и записи через интерфейс потоков. Пример — RLE: серии одинаковых символов заменяются на пары (символ, длина); кодирование — один проход с подсчётом серий и записью пар; декодирование — чтение пар и вывод символа n раз. Алгоритм использует только потоки и не требует хранения всех данных в памяти.
