## 4.1. Ассоциативная память, словарь. Варианты реализации, сравнение асимптотики

По учебнику (разделы 10.4, 10.6, 10.7.1, 11.1, 11.2): **словарь** (ассоциативный массив, коллекция «ключ–значение») — это структура данных, обеспечивающая доступ к информации **по ключу**, а не по позиции в коллекции. Термин **«ассоциативная память»** в литературе относят к **аппаратной реализации** этого принципа. Концепция словаря естественно вытекает из дерева поиска: по ключу можно найти соответствующий элемент; эта идея обобщается до отображения «ключ → значение» или «ключ → множество значений».

---

**Концепция словаря (по учебнику 11.1).** Ассоциативные массивы, или **словари**, предоставляют механизм доступа к данным **не по позиции**, а по **смысловой идентификации через ключ**. В общем виде ассоциативный массив — это **отображение** между множеством объектов **D** (значения, домен) и множеством **K** (ключи). Задаётся инъективное отображение **h : D → K** (разным элементам D соответствуют разные ключи из K); обратное отображение **h⁻¹** по ключу k возвращает единственный объект d (если ключ найден) или отсутствие значения. Поэтому результат поиска по ключу типизируют как **Option(D)**: либо найденное значение, либо «ключа нет». Если одному ключу может соответствовать **несколько элементов** (сюръективный случай: например, ключ «фамилия» → множество людей с этой фамилией), то значением служит **множество** (булеан D); формально **h⁻¹ : K → Option(D)** или **h⁻¹ : K → Option(β(D))**, где β(D) — множество всех подмножеств D. Словарь параметризуется **двумя типами**: типом ключей и типом элементов (значений), связываемых с ключами. В ряде источников структуру называют **map**. Типовые операции: **положить** пару (ключ, значение) — put/set; **получить** значение по ключу — get (возвращает Option или бросает исключение при отсутствии ключа); при необходимости — удаление по ключу, проверка наличия ключа.

---

**Реализация на хеш-таблице (по учебнику 11.2).** Самым непосредственным средством реализации словаря является **хеш-таблица**: она обеспечивает доступ к элементу по ключу за **константное время** в среднем. Это достигается тем, что **(а)** пары ключ–значение располагаются в **непрерывной области памяти** (аналогично массиву), **(б)** к ключам применяется **специальное преобразование** (хеш-функция), позволяющее сразу вычислить **положение элемента** в этой области. То есть по ключу вычисляется индекс (например, h(key) mod m), по индексу за O(1) выполняется доступ к ячейке (или к цепочке/пробе при коллизиях). **Вставка** и **поиск по ключу** в среднем **O(1)** при хорошей хеш-функции и разрешении коллизий (цепочки, открытая адресация). **Недостаток:** хеш-таблица **не хранит порядок** ключей — нет упорядоченного обхода по ключам и нет эффективных диапазонных запросов «все ключи от a до b».

---

**Реализация на дереве поиска (по учебнику 10.4, 10.6, 10.7.1).** **Дерево поиска** — конструкция, упорядочивающая элементы по ключу: задаётся множество элементов T и проекция (ключ) в множество K с **тотальным порядком**; в дереве поддерживается инвариант, позволяющий по ключу найти элемент за счёт сравнений. **Бинарное дерево поиска (BST):** в левом поддереве все ключи **меньше** ключа текущего узла, в правом — **больше** (предпочтительно строгие неравенства). Дерево по сути является формой словаря: по ключу можно найти соответствующее значение. **Преимущество** дерева — **логарифмическая сложность** операций вставки, удаления и поиска **O(log n)** при высоте дерева O(log n). Если элементы добавляются в дерево постоянно по возрастанию, дерево вырождается в линейный список, и асимптотика ухудшается. Поэтому применяют **балансировку**: **АВЛ-дерево**, **красно-чёрное дерево** (Red-Black tree) и др. Дерево считается сбалансированным, если высоты поддеревьев отличаются не более чем на заданную константу; при вставке и удалении выполняются **повороты** для восстановления баланса. **Плюсы** реализации словаря на дереве поиска: **упорядоченный обход** по ключам (симметричный, по возрастанию), **диапазонные запросы** «все пары с ключом от a до b» за O(log n + k), где k — число элементов в ответе; **минимальный и максимальный** ключ за O(log n) (или O(1), если хранить указатели на крайние узлы).

---

**Сравнение асимптотики.**

| Реализация | Вставка | Поиск по ключу | Упорядоченный обход / диапазоны |
|------------|---------|----------------|----------------------------------|
| Хеш-таблица | O(1) ср. | O(1) ср. | Нет |
| Дерево поиска (BST, красно-чёрное) | O(log n) | O(log n) | Да, O(n) обход, O(log n + k) диапазон |

**Итог.** Словарь (map, ассоциативная память) — структура «ключ → значение» (или ключ → множество значений); типовые операции put и get. **Два основных варианта реализации:** **(1)** **хеш-таблица** — O(1) в среднем вставка и поиск, порядок ключей не сохраняется; **(2)** **дерево поиска** (в т.ч. сбалансированное: АВЛ, красно-чёрное) — O(log n) вставка и поиск, упорядоченный обход и диапазонные запросы. Выбор зависит от того, нужны ли порядок ключей и диапазонные запросы (дерево) или достаточно быстрого доступа по ключу без порядка (хеш-таблица).

## 4.2. Полиморфный АТД «Множество»: реализация на базе хеш-таблиц

По учебнику (разделы 11.2, 11.3): **множество** — это совокупность элементов, в которой **порядок не специфицирован**. С математической точки зрения множество имеет чёткое определение, однако в контексте программной инженерии **нет единственного корректного способа** его реализации. Можно зафиксировать **абстрактный тип данных (АТД) «Множество»**, параметризуемый типом элементов **T** — **полиморфный АТД** (в учебнике приведён интерфейс **ISet&lt;T&gt;** с методами GetCount(), IsEmpty(), HasElement(T element), Union, Intersect, Except, Product). Рациональным решением является ограничение реализации **только конечными** множествами (бесконечные множества в общем случае трудно представить в памяти и реализовать операции над ними). Теоретически можно создать несколько реализаций множества, каждая на своей структуре данных: **массивы**, **хеш-таблицы** или **списки** (учебник, 11.3).

---

**Реализация множества на хеш-таблице.** Множество — коллекция **уникальных** элементов (каждый элемент встречается не более одного раза). Его можно реализовать как **словарь**, в котором **ключ совпадает с элементом**, а значение либо не хранится (достаточно наличия ключа), либо фиктивное (например, признак «присутствует»). Тогда **вставка** элемента — вставка пары (элемент, …) в хеш-таблицу; **проверка принадлежности** (contains / HasElement) — поиск по ключу в хеш-таблице. По учебнику 11.2 хеш-таблица обеспечивает доступ по ключу за **константное время** в среднем. Поэтому при реализации множества на хеш-таблице: **insert** (добавить элемент) — **O(1)** в среднем; **contains** (HasElement — проверить, входит ли элемент) — **O(1)** в среднем. **Удаление** элемента — удаление по ключу в хеш-таблице, также O(1) в среднем. Порядок элементов при обходе **не определён** (хеш-таблица не хранит порядок). Полиморфность: АТД параметризуется типом элементов T (ISet&lt;T&gt;); хеш-функция и сравнение на равенство должны быть определены для T.

**Пример кастомной реализации (хеш-таблица с цепочками):**

```cpp
#include <vector>
#include <list>
#include <functional>

template<typename T>
class HashSet {
    std::vector<std::list<T>> buckets_;
    size_t size_ = 0;
    std::hash<T> hasher_;
    static const size_t defaultCapacity = 16;

    size_t index(const T& x) const {
        return hasher_(x) % buckets_.size();
    }
public:
    HashSet() : buckets_(defaultCapacity) {}
    void insert(const T& x) {
        size_t i = index(x);
        for (const T& e : buckets_[i]) if (e == x) return;
        buckets_[i].push_back(x);
        ++size_;
    }
    bool contains(const T& x) const {
        size_t i = index(x);
        for (const T& e : buckets_[i]) if (e == x) return true;
        return false;
    }
    size_t getCount() const { return size_; }
};
```

---

**Итог 4.2.** Полиморфный АТД «Множество» (ISet&lt;T&gt;) можно реализовать на базе **хеш-таблицы**, отождествив элемент множества с ключом словаря. Вставка и проверка принадлежности — **O(1)** в среднем; порядок элементов не сохраняется. Подходит, когда важен быстрый поиск «есть ли элемент» и не нужен упорядоченный обход.

## 4.3. Полиморфный АТД «Множество»: реализация на базе дерева поиска (бинарного или B-дерева)

По учебнику (разделы 10.4, 10.6, 11.3): множество можно реализовать не только на хеш-таблице, но и на **дереве поиска** — при этом элементы множества хранятся как **ключи** в дереве поиска (значение при необходимости фиктивное или не хранится). Дерево поиска упорядочивает элементы по ключу и обеспечивает **логарифмическую** сложность вставки, удаления и поиска (учебник 10.4, 10.6).

---

**Реализация на бинарном дереве поиска (BST).** Элементы множества — ключи узлов дерева; в каждом узле хранится один элемент (при необходимости с балансировкой — АВЛ, красно-чёрное дерево). **Вставка** (insert) — вставка ключа в дерево поиска: спуск от корня по сравнению ключей, при отсутствии ключа создаётся новый узел. **Проверка принадлежности** (contains / HasElement) — поиск ключа в дереве: спуск от корня, сравнение с текущим узлом, переход в левое или правое поддерево. При сбалансированном дереве высота O(log n), поэтому **insert** и **contains** — **O(log n)**. **Плюс** по сравнению с хеш-таблицей: **упорядоченный обход** по элементам (симметричный обход дерева даёт элементы в порядке возрастания ключей); **минимальный и максимальный** элемент за O(log n) или O(1) при хранении указателей на крайние узлы; **диапазонные запросы** «все элементы от a до b» за O(log n + k).

---

**Реализация на B-дереве.** B-дерево (и B+-дерево) — сбалансированное многопутевое дерево поиска, оптимизированное для хранения на диске (узел — страница, много ключей в узле). Множество реализуется так же: элементы — ключи в дереве. **Вставка** и **поиск** — **O(log_t n)**, где t — порядок B-дерева, n — число элементов; при фиксированном t это O(log n). B-дерево применяют, когда данные не помещаются в оперативную память или нужна эффективная работа с большими объёмами с учётом блочного доступа.

**Пример кастомной реализации (множество на бинарном дереве поиска):**

```cpp
template<typename T>
class TreeSet {
    struct Node {
        T key;
        Node* left = nullptr;
        Node* right = nullptr;
        Node(const T& k) : key(k) {}
    };
    Node* root_ = nullptr;
    size_t size_ = 0;

    bool insert(Node*& n, const T& x) {
        if (!n) { n = new Node(x); ++size_; return true; }
        if (x < n->key) return insert(n->left, x);
        if (n->key < x) return insert(n->right, x);
        return false;  // уже есть
    }
    bool contains(Node* n, const T& x) const {
        if (!n) return false;
        if (x < n->key) return contains(n->left, x);
        if (n->key < x) return contains(n->right, x);
        return true;
    }
public:
    void insert(const T& x) { insert(root_, x); }
    bool contains(const T& x) const { return contains(root_, x); }
    size_t getCount() const { return size_; }
};
```

(Для гарантированного O(log n) нужна балансировка — АВЛ или красно-чёрное дерево; приведённый BST в худшем случае вырождается в список.)

---

**Сравнение с реализацией на хеш-таблице (4.2).** Дерево поиска (BST, B-дерево) даёт **O(log n)** вставку и проверку принадлежности вместо O(1) в среднем у хеш-таблицы, но сохраняет **порядок** элементов и поддерживает **упорядоченный обход** и диапазонные запросы. Выбор: хеш-таблица — когда достаточно быстрой проверки «принадлежит / не принадлежит» без порядка; дерево поиска — когда нужен порядок элементов, обход по возрастанию или запросы вида «все элементы в диапазоне».

---

**Итог 4.3.** Полиморфный АТД «Множество» (ISet&lt;T&gt;) можно реализовать на базе **дерева поиска** (бинарного BST или B-дерева), храня элементы как ключи. Вставка и проверка принадлежности — **O(log n)**; плюс **упорядоченный обход** по элементам и возможность диапазонных запросов. Тип элементов T должен поддерживать **тотальный порядок** (сравнение «меньше») для дерева поиска.

## 4.4. Организация пространства состояний в виде n-арного дерева: основные идеи (на примере задачи о рюкзаке)

По учебнику (раздел 12.2): **задача о рюкзаке** — классическая задача **дискретной оптимизации**: подобрать набор предметов с **максимальной стоимостью** при **ограничениях на ресурсы** (объём, вес, грузоподъёмность). **Организация пространства состояний в виде дерева** — основная идея: каждое **состояние** (набор уже выбранных предметов, текущий вес, объём, стоимость) представляется **вершиной** дерева; **переходы** между состояниями (добавление следующего предмета или отказ от него) — **рёбрами**; получается **дерево решений**, показывающее, как из начального состояния получаются все допустимые конфигурации.

---

**Формальная постановка (по учебнику 12.2).** Задано **множество предметов** {s₁, …, sₘ} (дискретное). Каждый предмет характеризуется параметрами: **вес** W, **стоимость** V, **объём** H. Ограничение на объём рюкзака: ∑ H(sᵢ) ≤ h₀. Цель: **max(∑ V(sᵢ))** при ограничениях (и при необходимости **min(∑ W(sᵢ))** при ограниченной вместимости). **Решение** — последовательность размещения предметов в рюкзак и **состояние рюкзака** (набор предметов, текущий вес, объём, суммарная стоимость). **Допустимое решение** — удовлетворяет всем ограничениям задачи.

---

**Дерево решений (по учебнику 12.2).** Идея: построить **дерево решений**, в котором **каждая вершина соответствует возможному состоянию рюкзака**. **Начальное состояние** — пустой рюкзак (корень дерева). **При добавлении каждого предмета** формируется **новое состояние**, отличающееся набором предметов, текущим весом, объёмом и суммарной стоимостью. **Из каждого состояния** возможен переход к **множеству новых состояний**, соответствующих **выбору следующего предмета** (в варианте 0/1 — два перехода: «взять предмет i» и «не взять»). Если новое состояние **нарушает ограничения** (например, превышает допустимый объём или вес), оно считается **недопустимым** и **исключается** из дальнейшего рассмотрения. Таким образом получается дерево, показывающее, каким образом выбранное состояние получено из начального. Если учитывать **форму предметов**, добавление одного элемента может порождать **не одно, а множество** новых состояний (разные способы размещения). Если **порядок размещения не фиксирован**, число комбинаций возрастает многократно — дерево **разрастается в ширину**, полный перебор всех состояний из-за ограничений по памяти и времени **невозможен**. На практике часто используют **обход в глубину** (DFS), позволяющий быстрее находить допустимое (пусть не оптимальное) решение. Дальнейшее улучшение достигается методами **динамического программирования** или **жадных алгоритмов** (учебник 12.2).

---

**Рюкзак 0/1 (частный случай).** n предметов, вес w[i], ценность v[i], вместимость W. Состояние задаётся парой (i, weight): рассмотрены предметы 0..i−1, текущий вес weight. Два перехода из состояния (i, w): «не взять предмет i» → (i+1, w); «взять предмет i» → (i+1, w + w[i]) при условии w + w[i] ≤ W (иначе ветвь отсекается). Листья — состояния (n, ·). Рекурсия: значение в состоянии (i, w) = max(значение(i+1, w), v[i] + значение(i+1, w+w[i])) при w+w[i] ≤ W; иначе значение(i+1, w). Число вершин дерева в худшем случае до 2ⁿ (без отсечения недопустимых и без мемоизации). С мемоизацией или динамическим программированием по (i, w) сложность O(n·W).

**Пример решения (нисходящее ДП с мемоизацией по (i, w)):**

```cpp
#include <vector>
#include <unordered_map>
#include <algorithm>

// Рюкзак 0/1: n предметов, веса w, ценности v, вместимость W.
// Возвращает максимальную суммарную ценность.
int knapsack(int i, int weight, const std::vector<int>& w, const std::vector<int>& v, int W,
             std::unordered_map<int, int>& memo, int n) {
    if (i == n) return 0;  // лист дерева — рассмотрели все предметы
    int key = i * (W + 1) + weight;  // одномерный ключ для (i, weight)
    if (memo.count(key)) return memo[key];
    int skip = knapsack(i + 1, weight, w, v, W, memo, n);  // не взять предмет i
    int take = (weight + w[i] <= W)
        ? v[i] + knapsack(i + 1, weight + w[i], w, v, W, memo, n)  // взять предмет i
        : -1;
    memo[key] = (take >= 0) ? std::max(skip, take) : skip;
    return memo[key];
}
// Вызов: unordered_map<int,int> memo; int ans = knapsack(0, 0, w, v, W, memo, n);
```

(Восходящее ДП: таблица `dp[i][w]` — макс. ценность для предметов 0..i−1 и вместимости w; заполнять по возрастанию i и w.)

**Итог 4.4.** Организация пространства состояний в виде **n-арного дерева** (на примере рюкзака): вершина — **состояние** (набор выбранных предметов, вес, объём, стоимость); рёбра — **переходы** (добавление предмета или отказ); недопустимые состояния отсекаются; дерево может экспоненциально разрастаться; на практике — обход в глубину, динамическое программирование или жадные алгоритмы.

## 4.5. Организация пространства состояний в виде n-арного дерева: основные идеи (на примере минимаксного алгоритма)

**Организация пространства состояний в виде дерева** применяется не только в задачах оптимизации (рюкзак), но и в **игровом поиске**: пространство **позиций игры** и **ходов** естественно представляется **деревом**: вершины — **позиции** (состояния игры), рёбра — **ходы** (переход из позиции в позицию). Корень — начальная позиция; сыновья вершины — позиции, достижимые за один ход; листья — терминальные позиции (конец игры) или позиции, до которых ограничиваются поиском (с эвристической оценкой). Такое дерево называют **деревом игры** (game tree). **Минимаксный алгоритм** — способ приписать каждой позиции **оценку** в предположении, что оба игрока играют оптимально, и на этой основе выбрать лучший ход.

---

**Минимакс.** Предполагается игра **двух игроков**: один стремится **максимизировать** оценку (Max), другой — **минимизировать** (Min); ходы чередуются. В **листьях** дерева задаётся оценка: в терминальных позициях — результат игры (например, +1 победа первого, −1 победа второго, 0 ничья); в неразвёрнутых — **эвристическая оценка** позиции. Рекурсивное определение оценки вершины: если ход **Max** — значение вершины = **max** по значениям детей; если ход **Min** — значение вершины = **min** по значениям детей. Таким образом, в корне получается оценка позиции при оптимальной игре обеих сторон; **лучший ход** для текущего игрока — переход в ребёнка с оптимальным значением (max для Max, min для Min). Обход дерева — обычно **в глубину** (DFS); оценка вершины вычисляется после оценки всех её детей. Сложность полного перебора — экспоненциальна по глубине дерева (число вершин может быть порядка b^d, где b — ветвящесть, d — глубина).

---

**Альфа-бета отсечение (alpha-beta pruning).** Чтобы не просматривать все вершины дерева, применяют **отсечение**: если при обходе детей вершины уже получена оценка, при которой **родительская вершина** (игрок на уровне выше) **не выберет** эту ветвь, остальных детей этой вершины можно **не рассматривать**. Формально: при обходе поддерживают **α** (нижняя граница для Max) и **β** (верхняя граница для Min). При вычислении значения в вершине Max: если значение одного из детей уже ≥ β, то Min в родителе не выберет эту ветвь (ему достаточно β), поэтому остальных детей этой вершины не просматриваем. Аналогично для вершины Min: если значение ребёнка уже ≤ α, отсекаем остальных детей. При удачном порядке просмотра ходов альфа-бета позволяет сократить число просматриваемых вершин (в лучшем случае — до порядка b^(d/2)), не меняя результата минимакса.

**Пример решения (минимакс и альфа-бета).** Предполагается интерфейс позиции: терминальность, оценка в листе, список ходов и переход в новую позицию. Ход Max — максимизируем, ход Min — минимизируем.

```cpp
// Минимакс: возвращает оценку позиции (Max стремится к +∞, Min к −∞).
int minimax(const Position& pos, bool isMaxTurn) {
    if (pos.isTerminal()) return pos.evaluate();  // лист дерева
    auto moves = pos.getMoves();
    if (isMaxTurn) {
        int best = -INF;
        for (const auto& m : moves) {
            int val = minimax(pos.makeMove(m), false);
            best = std::max(best, val);
        }
        return best;
    } else {
        int best = +INF;
        for (const auto& m : moves) {
            int val = minimax(pos.makeMove(m), true);
            best = std::min(best, val);
        }
        return best;
    }
}

// Альфа-бета отсечение: α — нижняя граница для Max, β — верхняя для Min.
int alphabeta(const Position& pos, int depth, int alpha, int beta, bool isMaxTurn) {
    if (depth == 0 || pos.isTerminal()) return pos.evaluate();
    auto moves = pos.getMoves();
    if (isMaxTurn) {
        int v = -INF;
        for (const auto& m : moves) {
            v = std::max(v, alphabeta(pos.makeMove(m), depth - 1, alpha, beta, false));
            alpha = std::max(alpha, v);
            if (beta <= alpha) break;  // отсечение
        }
        return v;
    } else {
        int v = +INF;
        for (const auto& m : moves) {
            v = std::min(v, alphabeta(pos.makeMove(m), depth - 1, alpha, beta, true));
            beta = std::min(beta, v);
            if (beta <= alpha) break;  // отсечение
        }
        return v;
    }
}
```

(В реальной игре вызывают alphabeta для каждого возможного хода и выбирают ход с лучшей оценкой; глубина depth ограничивает поиск.)

---

**Итог 4.5.** Организация пространства состояний в виде **n-арного дерева** (на примере минимакса): вершины — **позиции в игре**, рёбра — **ходы**; листья — терминальные позиции или позиции с эвристической оценкой. **Минимакс**: Max выбирает **max** по детям, Min — **min** по детям; оценка корня — результат при оптимальной игре. **Альфа-бета отсечение** — не просматривать ветви, которые не могут повлиять на выбор родителя; уменьшает число просматриваемых вершин при сохранении результата.

## 4.6. Ускорение поиска с помощью кеширования

По учебнику (раздел 13.3, 11.5): **ускорение поиска с помощью кеширования** — сохранение уже полученных результатов (частичных решений, значений подзадач или часто запрашиваемых данных), чтобы при повторном обращении не вычислять их заново. В контексте рекурсивных алгоритмов и дискретной оптимизации это реализуется как **мемоизация**; в контексте доступа к данным — как **кэширование** (см. 4.7 и учебник 11.5). Ниже — мемоизация и динамическое программирование (учебник 13.3).

---

**Мемоизация.** **Мемоизация** — приём, при котором **результаты вычислений по аргументам сохраняются** (в кеше, таблице, словаре). При повторном вызове функции с теми же аргументами **возвращается сохранённое значение**, а не выполняется повторное вычисление. Так **избегают повторных вычислений** одних и тех же подзадач. В рекурсивных алгоритмах (например, рекурсия по состояниям в задаче о рюкзаке или в графе состояний) без мемоизации одна и та же подзадача может вычисляться экспоненциальное число раз; с мемоизацией каждое состояние (набор аргументов) вычисляется **не более одного раза**. Это превращает **экспоненциальную** по времени рекурсию в **полиномиальную** (или псевдополиномиальную, в зависимости от задачи), за счёт дополнительной памяти на хранение результатов.

---

**Динамическое программирование (по учебнику 13.3).** **Динамическое программирование (ДП)** — обобщённая методика эффективного решения **дискретных оптимизационных задач**, многие из которых сводятся к поиску **оптимального пути в графе состояний**. **Основной принцип ДП** — **принцип оптимальности Беллмана**: оптимальный путь обладает тем свойством, что **какова бы ни была последняя вершина на этом пути, предшествующий ей участок также является оптимальным путём** до этой вершины. Это позволяет **разбить сложную задачу на более простые подзадачи** и **избежать повторных вычислений**, сохраняя уже найденные оптимальные решения для подзадач (**мемоизация**). Вершины графа при этом представляют **состояния** системы (дискретные, комбинационного характера).

**Два основных подхода к ДП (учебник 13.3):**

1. **Нисходящий процесс (сверху вниз):** решение основной задачи разбивается на подзадачи, те — на ещё более мелкие, до тривиальных случаев. Реализуется **рекурсией**. Соответствует обходу графа **в глубину**. Мемоизация — при первом вычислении результата для подзадачи он записывается в таблицу/словарь; при повторном обращении к той же подзадаче результат берётся из таблицы.

2. **Восходящий процесс (снизу вверх):** сначала решаются **все самые мелкие** подзадачи, затем на их основе — подзадачи более крупного уровня, и так далее, пока не будет решена исходная задача. Реализуется **итеративно** (циклы, заполнение таблицы). Соответствует обходу графа **в ширину**. Таблица заполняется в порядке возрастания «размера» подзадачи.

На практике восходящий подход часто удобнее (нет рекурсивных вызовов, порядок заполнения явный); нисходящий с мемоизацией может вычислять только те подзадачи, которые реально достигаются из начального состояния.

---

**Итог 4.6.** **Кеширование** (в широком смысле) ускоряет поиск за счёт сохранения уже полученных результатов. **Мемоизация** — кеш результатов по аргументам (подзадачам); при повторном обращении — возврат из кеша; превращает экспоненциальную рекурсию в полиномиальную. **Динамическое программирование** — методика решения оптимизационных задач с использованием принципа оптимальности Беллмана и **мемоизации частичных результатов**; реализуется нисходяще (рекурсия + мемоизация) или восходяще (итеративное заполнение таблицы).

## 4.7. Ускорение поиска с помощью индексирования

По учебнику (раздел 11.5): **индекс** — специальная структура (типа **ассоциативная память**), **ускоряющая поиск** информационных объектов **по ключу** (значению атрибутов). Для построения индекса необходимо заранее зафиксировать структуру данных и перечень атрибутов, по которым допускается выполнять поиск. Над индексируемой коллекцией **недопустимо произвольное** выполнение сортировки или перестройки элементов — **индекс должен оставаться согласованным** с исходными данными.

---

**Зачем нужен индекс (по учебнику 11.5).** Исходная коллекция может храниться где угодно (массив, список, файл). Без дополнительной структуры элементы располагаются в случайном порядке, и **поиск элемента** (по атрибуту) имеет трудоёмкость **O(n)**. В системах, где поиск по атрибутам — основная операция, линейный обход недопустимо медленный. Простое решение — отсортировать коллекцию по атрибуту и использовать **бинарный поиск** O(log n). Но у этого подхода недостатки: первичная сортировка больших данных затратна; коллекция динамична (вставки и удаления); вставка в отсортированную структуру требует сдвига; поиск может выполняться по **разным** атрибутам, поэтому заранее выбрать единственный атрибут сортировки невозможно. **Решение** — построение **дерева поиска** (или другой структуры индекса) **по выбранному атрибуту**.

---

**Реализация индекса (по учебнику 11.5).** Индекс по выбранному атрибуту строится как **дерево поиска** (или хеш-таблица). Если атрибут **однозначно идентифицирует** объект (ключевой атрибут), в узлах дерева хранятся пары **(ключ, ссылка на объект)**. Если атрибут **неключевой** (например, год рождения, фамилия), нескольким объектам может соответствовать одно значение — в дереве хранятся **списки объектов** с одинаковым ключом. Таким образом, индексирование делится на два типа: **по ключевым атрибутам** (результат поиска единственный) и **по неключевым атрибутам** (результат — множество объектов). **B+-дерево** в базах данных — типичная структура индекса: O(log n) поиск по ключу, поддержка **диапазонных запросов** (все записи с ключом от a до b) и эффективный последовательный обход по ключам; данные в листьях, внутренние узлы — разделители для маршрутизации.

---

**Обратный индекс.** **Обратный индекс** (inverted index) — вид индекса, в котором ключ — это **термин** (например, слово), а значение — **список документов** (или позиций), в которых этот термин встречается. То есть отображение **слово → список документов**. Используется в полнотекстовом поиске: по запросу (набору слов) быстро находятся все документы, содержащие эти слова. Реализуется как словарь (хеш-таблица или дерево поиска), где ключ — слово, значение — список идентификаторов документов или список вхождений.

---

**Сравнение с кешированием (учебник 11.5).** **Кэширование** преследует ту же цель — ускорение доступа к данным, но работает иначе: кэш изначально **пуст**, по мере обращений заполняется запрошенными объектами; при переполнении удаляются элементы, к которым долго не обращались. **Индекс** уместен, когда имеется **коллекция объектов** и нужен быстрый поиск по атрибутам. **Кэширование** полезно, когда операций поиска/доступа много и есть различия в **частоте обращения** к объектам; особенно эффективно, когда прямое получение данных дорого (диск, внешняя система).

---

**Итог 4.7.** **Индекс** — вспомогательная структура (ассоциативная память) для быстрого поиска по ключу (атрибуту); должна оставаться согласованной с коллекцией. Реализация — дерево поиска (в т.ч. **B+-дерево** в БД) или хеш-таблица; поиск O(log n) или O(1) в среднем; B+-дерево даёт также диапазонные запросы. **Обратный индекс** — слово → список документов, для полнотекстового поиска. Отличие от кеширования: индекс строится по коллекции и атрибутам; кэш заполняется по мере обращений и может вытеснять редко используемые данные.

## 4.8. Сравнительная характеристика систем: реального времени, онлайн, асинхронных

**Системы реального времени (real-time), онлайн-системы (online)** и **асинхронные системы** — три разных способа организации обработки данных и отклика; выбор зависит от требований к задержкам, полноте входных данных и модели выполнения.

---

**Системы реального времени (real-time).** Требуют **гарантированного отклика в заданные сроки** (deadline). Результат считается корректным только если он получен вовремя; опоздание равносильно ошибке (в жёстком real-time — катастрофа: АСУ ТП, тормоза, автопилот). Различают **жёсткое real-time (hard)** — нарушение срока недопустимо; **мягкое (soft)** — редкие опоздания допустимы, но снижают качество (например, пропуск кадра в видео). Типичные области: управление (регуляторы, датчики), встраиваемые системы (embedded), телеком, медицина. Реализация опирается на планировщики с приоритетами, ограничение времени выполнения, предсказуемые структуры данных (избегают динамической аллокации в критическом пути). **Поиск и данные:** доступ к данным и вычисления должны укладываться в окно времени; часто используются фиксированные буферы и индексы с гарантированной сложностью O(log n) или O(1).

---

**Онлайн-системы и онлайн-алгоритмы (online).** **Данные поступают потоком**; решение принимается **без полного знания входа** — каждый следующий элемент потока обрабатывается по мере поступления, без возможности «заглянуть вперёд» или переобработать прошлое. Противопоставление: **офлайн** — весь вход известен заранее, можно сортировать, планировать, оптимизировать глобально; **онлайн** — только уже увиденный префикс. Примеры: стриминг (обработка видеопотока, логов), поточные агрегаты (скользящее среднее, топ-k за последние N элементов), адаптивные структуры (добавление в индекс по мере прихода записей). Качество онлайн-алгоритма оценивают через **competitive ratio** — отношение результата онлайн-алгоритма к оптимальному офлайн-решению. **Поиск:** индекс строится или обновляется по мере поступления данных; запросы обслуживаются по текущему снимку (B+-дерево, хеш-таблица с инкрементальными вставками).

Пример: подсчёт скользящей суммы по потоку чисел (онлайн) — решение без полного входа.

```cpp
// Онлайн: обрабатываем поток, не храня весь вход
struct OnlineSum {
    long long sum = 0;
    void add(int x) { sum += x; }  // один элемент потока
    long long get() const { return sum; }
};
// Скользящее окно размера k — только последние k элементов влияют
struct SlidingWindowSum {
    std::queue<int> q;
    long long sum = 0;
    int k;
    void add(int x) {
        if ((int)q.size() == k) { sum -= q.front(); q.pop(); }
        q.push(x); sum += x;
    }
    long long get() const { return sum; }
};
```

---

**Асинхронные системы.** **Вызов операции не блокирует** поток выполнения: вместо ожидания результата вызывающий код продолжает работу, а **результат получают позже** — через **callback** (функция обратного вызова) или **future/promise** (деferred value). Цель — не простаивать на I/O или тяжёлых вычислениях; параллельное выполнение многих операций без создания избыточного числа потоков. Типичное применение: сетевые запросы, чтение с диска, параллельные вычисления. **Поиск:** асинхронный запрос к индексу или к БД — запрос отправляется, поток не блокируется; по готовности вызывается callback или считывается future.

Пример: асинхронный вызов в C++ (std::async, std::future) — вызов не блокирует; результат забирается позже.

```cpp
#include <future>
#include <chrono>

// Асинхронный поиск: не блокируем поток, результат — в future
std::future<int> async_search(const std::vector<int>& data, int key) {
    return std::async(std::launch::async, [&data, key]() {
        for (size_t i = 0; i < data.size(); ++i)
            if (data[i] == key) return (int)i;
        return -1;
    });
}

// Использование: делаем другие дела, потом get() — блокируем только при получении результата
auto f = async_search(vec, 42);
// ... параллельно можно выполнять другую работу ...
int index = f.get();  // здесь ждём результат, если ещё не готов
```

Вариант с **callback** (тот же смысл — не блокировать на время поиска):

```cpp
template<typename Callback>
void search_async(const std::vector<int>& data, int key, Callback on_done) {
    std::thread([&data, key, on_done]() {
        int result = -1;
        for (size_t i = 0; i < data.size(); ++i)
            if (data[i] == key) { result = (int)i; break; }
        on_done(result);
    }).detach();
}
// Вызов: search_async(vec, 42, [](int idx) { /* обработать результат */ });
```

---

**Сравнительная таблица.**

| Критерий | Real-time | Online | Асинхронные |
|----------|-----------|--------|-------------|
| Главное требование | Отклик к deadline | Решение без полного входа | Вызов не блокирует поток |
| Входные данные | Часто поток датчителей/событий | Поток, префикс за префиксом | Любой; запрос «отложен» |
| Типичные области | Управление, embedded, телеком | Стриминг, поточная аналитика | Сеть, диск, параллельные задачи |
| Модель результата | Ответ вовремя или сбой | Агрегат/решение по текущему префиксу | Результат через callback/future |

---

**Итог 4.8.** **Real-time** — системы с жёсткими или мягкими сроками отклика; корректность зависит от соблюдения deadline. **Online** — данные потоком, решения и поиск без полного входа; индекс и структуры обновляются по мере поступления данных. **Асинхронные** — неблокирующие вызовы; результат получают через callback или future, что позволяет совмещать много операций без простоя потока. Три подхода можно комбинировать: например, асинхронный онлайн-обработчик потока с ограничениями по задержке (soft real-time).

## 4.9. Разреженные структуры: одномерные

**Разреженный одномерный массив** — вектор, в котором **большинство элементов нулевые** (или совпадают с одним и тем же «пустым» значением). Хранить все n ячеек неэффективно по памяти и по времени доступа, если ненулевых элементов мало: **k ≪ n**. Идея: хранить только **пары (индекс, значение)** для ненулевых позиций; доступ по индексу — поиск по ключу.

**Реализация на дереве поиска.** Ключ — индекс, значение — элемент. Вставка, поиск, удаление — **O(log k)**, где k — число ненулевых. Упорядоченный обход по индексам возможен (симметричный обход дерева). Подходит, когда нужен обход по возрастанию индекса или диапазонные запросы.

**Реализация на словаре (хеш-таблица).** Ключ — индекс, значение — элемент. Вставка и доступ по индексу — **O(1)** в среднем. Порядок индексов не сохраняется; обход по всем ненулевым — за O(k). Выбор: дерево — если нужен порядок или диапазоны; хеш — если достаточно случайного доступа по индексу.

```cpp
// Разреженный вектор: храним только ненулевые (индекс → значение)
#include <unordered_map>
std::unordered_map<int, double> sparse_vec;  // индекс → значение
sparse_vec[1000] = 3.14;   // O(1) ср., не создаём 1000 ячеек
double x = sparse_vec[1000];  // 3.14; отсутствующий ключ даёт 0 или нужно проверять
```

**Итог 4.9.** Разреженный одномерный массив — храним только ненулевые элементы как пары (индекс, значение). Дерево поиска: O(log k), порядок индексов. Хеш-таблица: O(1) ср., без порядка. k — число ненулевых.

## 4.10. Разреженные структуры: многомерные

**Разреженная многомерная структура** — матрица или массив большой размерности, в которых **большинство элементов нулевые**. Хранить полный массив размера n₁×n₂×… неэффективно; храним только **ненулевые** элементы, идентифицируемые **составным ключом** — кортежем индексов (i, j) для матрицы или (i₁, i₂, …, iₙ) для n-мерного массива.

**Реализация на ассоциативной памяти.** Ключ — **пара (i, j)** или кортеж индексов; значение — элемент. Используют словарь (дерево поиска или хеш-таблицу): ключ — упорядоченная пара или хеш от кортежа. **Дерево поиска** (ключ — лексикографически (i, j)): доступ O(log k), возможны диапазонные запросы по подматрице. **Хеш-таблица** (ключ — (i, j), хеш-функция от пары): доступ **O(1)** в среднем. Память — O(k), где k — число ненулевых элементов.

**Итог 4.10.** Разреженные многомерные данные хранят только ненулевые элементы; ключ — кортеж индексов (i, j) или (i₁, …, iₙ). Реализация — словарь (дерево или хеш): O(log k) или O(1) ср. доступ, память O(k).

## 4.11. Пространственные данные: одномерный случай. Дерево отрезков

**Одномерные пространственные (интервальные) данные** — массив, по которому нужны **запросы на отрезке [l, r]** (сумма, минимум, максимум и т.п.) и **точечные обновления** (изменить элемент по индексу). Наивно: запрос за O(n), обновление O(1). **Дерево отрезков (segment tree)** даёт **O(log n)** и на запрос, и на обновление при O(n) памяти.

**Структура.** Полное бинарное дерево: **листья** соответствуют элементам массива (один лист — один индекс); **внутренний узел** хранит **агрегат** по отрезку своих детей (сумму, min, max и т.д.). Корень отвечает за весь отрезок [0, n−1]; узел v с отрезком [tl, tr] имеет детей с отрезками [tl, tm] и [tm+1, tr]. Массив дерева часто хранят в виде heap: вершина v — дети 2v и 2v+1.

**Запрос на отрезке [l, r].** Рекурсивно спускаемся от корня: если текущий отрезок узла [tl, tr] не пересекается с [l, r], возвращаем нейтраль (0 для суммы, +∞ для min); если [tl, tr] целиком внутри [l, r], возвращаем значение узла; иначе делим отрезок пополам и объединяем результаты детей. На каждом уровне рекурсии обрабатывается O(1) узлов, уровней O(log n) — **итого O(log n)**.

**Точечное обновление.** Спуск от корня до листа по индексу i, обновление листа, затем **пересчёт вверх** всех узлов на пути от листа к корню. Глубина O(log n) — **O(log n)** время.

**Пример: запрос суммы на отрезке.**

```cpp
// tree[] — массив дерева отрезков (сумма), n — размер исходного массива
int query(int v, int tl, int tr, int l, int r) {
    if (l > tr || r < tl) return 0;           // нет пересечения — нейтраль
    if (l <= tl && tr <= r) return tree[v];   // узел целиком в [l, r]
    int tm = (tl + tr) / 2;
    return query(2*v, tl, tm, l, r) + query(2*v+1, tm+1, tr, l, r);
}

void update(int v, int tl, int tr, int pos, int delta) {
    if (tl == tr) { tree[v] += delta; return; }
    int tm = (tl + tr) / 2;
    if (pos <= tm) update(2*v, tl, tm, pos, delta);
    else update(2*v+1, tm+1, tr, pos, delta);
    tree[v] = tree[2*v] + tree[2*v+1];
}
```

**Итог 4.11.** Дерево отрезков — структура для одномерного массива с запросами на отрезке (сумма, min, max) и точечными обновлениями. Полное бинарное дерево: листья — элементы, узел — агрегат по отрезку. Запрос и обновление — **O(log n)**, память O(n).
