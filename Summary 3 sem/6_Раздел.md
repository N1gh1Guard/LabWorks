## 6.1. Указатель как идиома и варианты реализации через индексирование коллекций и с помощью итераторов

По учебнику (разделы 5.10.1, 8.2): **идиома** — концепция, приём, общая идея, которая может реализовываться разными способами. **Пример: указатель как идиома.** Это переменная, значение которой — **адрес объекта** определённого типа (в C может быть void*). В абстрактном виде указатель — нечто, поддерживающее: **(1)** возможность **указывать на элемент** в известной области данных; **(2)** возможность **обратиться** к этому элементу; **(3)** **арифметику указателей** — перемещение по области данных. **Реализация указателя как типа данных — лишь один из вариантов** реализации этой идиомы. **Другими примерами являются итератор и индекс в массиве**, поскольку они тоже используются как инструмент для доступа к элементу в известной области и перемещения по ней (учебник 5.10.1).

**Указатель (как тип в C/C++).** По учебнику 8.2 указатель — **численное значение**, обозначающее ячейку памяти, с которой начинается объект. При **перераспределении памяти** (realloc, перемещение контейнера) адрес может измениться — указатель **инвалидируется**. Требуется явное обновление или использование другого механизма доступа.

**Индекс.** Целое число — **позиция элемента в контейнере** (например, в массиве или списке). При перемещении данных в памяти индекс сам по себе не меняется; обращение к элементу выполняется через контейнер: `container[index]`. **Остаётся корректным**, пока контейнер и индекс в допустимых границах; не инвалидируется при realloc, если контейнер обновляет хранилище, а доступ идёт через интерфейс контейнера.

**Итератор.** По учебнику (раздел 13.2) **итератор** — объект, который позволяет **последовательно обходить элементы коллекции**, не раскрывая её внутреннего устройства; инкапсулирует логику перехода к следующему элементу. Абстракция **позиции** в коллекции: операции `++`, `*` (разыменование), `!=` (сравнение с концом). Может быть реализацией идиомы «указатель» для данной коллекции (учебник 5.10.1).

```cpp
// Указатель: адрес, инвалидируется при realloc
int* p = arr;  // p указывает на arr[0]; после realloc(arr) p может быть неверен

// Индекс: позиция, остаётся корректным при перемещении контейнера
size_t i = 0;  // доступ через v[i]; контейнер сам знает, где данные

// Итератор: абстракция позиции, ++, *, !=
for (auto it = v.begin(); it != v.end(); ++it) { int x = *it; }
```

**Итог 6.1.** По учебнику 5.10.1 **указатель как идиома** — доступ к элементу в области данных и перемещение по ней; реализациями служат **указатель** (тип в C/C++), **индекс** в коллекции и **итератор**. Указатель инвалидируется при realloc; индекс и итератор привязаны к контейнеру и остаются корректными при корректном использовании.

## 6.2. Арифметика указателей в C/C++

По учебнику (раздел 8.2): для указателя **определены операции сложения и вычитания**. При прибавлении целого числа n к указателю он будет указывать на ячейку, адрес которой равен **текущее значение + sizeof(тип, на который указывает указатель) × n** (учебник 8.2).

**Операции.** **p + k** — сдвиг на k элементов вперёд (адрес увеличивается на k × sizeof(*p)). **p - k** — сдвиг назад. **p - q** — разность двух указателей в **элементах** (результат типа ptrdiff_t); определена только для указателей в одну область памяти. ***(p + k)** эквивалентно **p[k]** — доступ к элементу на расстоянии k от p.

**Итераторы.** Для **Random Access** итераторов (vector, array, указатель) поддерживается арифметика: `it + k`, `it - k`, `it - it2` — аналогично указателям. Для **Bidirectional** итераторов (list, set) — только `++`, `--`; сдвига на k нет.

```cpp
int arr[10];
int* p = arr;
*(p + 5) == p[5];   // один и тот же элемент
int* q = p + 7;
ptrdiff_t d = q - p;  // d == 7

std::vector<int> v(10);
auto it = v.begin();
*(it + 3) == it[3];  // Random Access
// std::list::iterator — только ++it, --it
```

**Итог 6.2.** По учебнику 8.2 **арифметика указателей**: p+k — сдвиг на k элементов (адрес + k×sizeof(тип)); p−q — разность в элементах; *(p+k) == p[k]. Для итераторов: Random Access — it+k, it−it2; Bidirectional — только ++, --.

## 6.3. Переопределение операторов в C++

По учебнику (раздел 6.7): C++ позволяет **переопределять операторы** для пользовательских типов (классов и структур), чтобы обеспечить интуитивное и естественное поведение при использовании стандартных операций (+, -, ==, [], () и др.) и сделать пользовательские типы похожими на встроенные по поведению (учебник 6.7).

**Функция-член класса.** Левый операнд — **this**; правый передаётся параметром. Например, `a + b` вызывается как `a.operator+(b)`. Подходит, когда левый операнд всегда принадлежит нашему типу.

**Свободная (внешняя) функция.** Оба операнда передаются параметрами. Нужна для **симметрии**: если определён только член `operator+`, то `a + b` работает, а `k * v` (скаляр × вектор), когда левый операнд — не наш класс, — нет. По учебнику 6.7: «если a+b работает, то желательно, чтобы b+a также имел смысл — для этого стоит писать внешние перегрузки с friend или шаблонные».

**Семантика.** По учебнику 6.7 **важно, чтобы оператор отражал семантику операции**: если используется +, то результат **не должен изменять операнды**. Оператор **operator=** требует аккуратного подхода (работа с указателями, проверка **самоприсваивания**). Некоторые операторы язык не позволяет перегружать: `.`, `::`, `sizeof`, `typeid` (учебник 6.7).

```cpp
class Vector {
    double x, y;
public:
    Vector operator+(const Vector& other) const {  // член: a + b
        return Vector(x + other.x, y + other.y);
    }
};
// Свободная функция для симметрии: k * v
Vector operator*(double k, const Vector& v) {
    return Vector(k * v.x(), k * v.y());
}
```

**Итог 6.3.** По учебнику 6.7 переопределение операторов — функция-член (левый операнд = this) или свободная (для симметрии, например k*v). Оператор не должен изменять операнды там, где это не предполагается; operator= — член класса, с учётом самоприсваивания. Часть операторов перегружать нельзя.

## 6.4. Особенности переопределения операторов new и delete

**Цепочка вызовов.** Выражение **new T** (или new T(args)) выполняет: **(1)** выделение памяти — вызывается **operator new**(size) (или operator new[](size) для массивов); **(2)** на полученном адресе вызывается **конструктор** T (или T(args)). Выражение **delete p** выполняет: **(1)** вызов **деструктора** объекта по адресу p; **(2)** освобождение памяти — **operator delete**(p) (или operator delete[] для массивов).

**Перегрузка.** **operator new** и **operator delete** можно перегружать **глобально** (влияет на всё выделение в программе) или **для класса** (только для объектов этого класса). Типичные цели: **пулы памяти** (выделение блоками, быстрый alloc/free), **отладка** (логирование, проверка утечек), выравнивание или особая политика размещения.

**Сигнатуры.** Стандартные формы: `void* operator new(size_t size);` `void operator delete(void* p);` для массивов — `operator new[]`, `operator delete[]`. Есть формы с дополнительными параметрами (placement new, nothrow и др.).

```cpp
// Перегрузка для класса: пул или отладка
struct MyClass {
    static void* operator new(size_t size) {
        std::cout << "alloc " << size << "\n";
        return ::operator new(size);  // глобальный new
    }
    static void operator delete(void* p) {
        std::cout << "free\n";
        ::operator delete(p);
    }
};
```

**Итог 6.4.** **new T** → operator new (выделение) → конструктор; **delete p** → деструктор → operator delete. Перегрузка operator new/delete — глобально или для класса; применяется для пулов памяти, отладки и специальных политик выделения.

## 6.5. Умные указатели

По учебнику (раздел 8.6): при работе с указателями возникают проблемы **управления памятью** — объект создаётся в одном блоке, используется в нескольких, и в какой-то момент должен быть уничтожен. Подход в C (уничтожать там, где создан, передавать по указателю) порождает побочные эффекты и затрудняет композицию. **Идея умных указателей:** умный указатель **хранит объект** (или ссылку на него) и **понимает, когда значение больше не нужно**; для этого достаточно **подсчёта ссылок** на хранимый объект. Умные указатели позволяют **автоматически управлять жизненным циклом** объекта (учебник 8.6).

**unique_ptr.** По учебнику 8.6 **unique_ptr** — указатель, который **единолично владеет** объектом. Используется, когда нужно передать объект **по значению** (передача владения), а не по ссылке. При уничтожении unique_ptr вызывается delete для объекта; копирование запрещено, допускается только **перемещение** (std::move). **RAII:** ресурс (память) привязан к времени жизни объекта указателя.

**shared_ptr.** По учебнику 8.6 в стандартной библиотеке C++ умный указатель с подсчётом ссылок представлен типом **shared_ptr**. Несколько shared_ptr могут указывать на один объект; счётчик ссылок увеличивается при копировании и уменьшается при уничтожении копии; когда счётчик становится нулём, объект удаляется. Удобен для **разделяемого владения**, но при **циклических ссылках** (A→B→A) счётчики не обнуляются — объекты не освобождаются (учебник 8.6).

**weak_ptr.** По учебнику 8.6 **weak_ptr** ведёт себя аналогично shared_ptr, но **не увеличивает счётчик ссылок**. Нужен для **решения проблемы циклических зависимостей**: если A и B ссылаются друг на друга через shared_ptr, цикл не разорвать; если одна из ссылок — weak_ptr, она не удерживает объект, и при уничтожении «сильных» ссылок объекты освобождаются. Для доступа к объекту из weak_ptr получают временный shared_ptr (lock()).

```cpp
#include <memory>

// unique_ptr: единоличное владение, RAII
std::unique_ptr<int> p = std::make_unique<int>(42);
// std::unique_ptr<int> q = p;  // ошибка: копирование запрещено
std::unique_ptr<int> q = std::move(p);  // владение передано

// shared_ptr: подсчёт ссылок
std::shared_ptr<int> s1 = std::make_shared<int>(1);
std::shared_ptr<int> s2 = s1;  // счётчик == 2

// weak_ptr: не увеличивает счётчик, разрыв циклов
std::weak_ptr<int> w = s1;
if (auto locked = w.lock()) { /* использовать *locked */ }
```

**Итог 6.5.** По учебнику 8.6 **умные указатели** автоматизируют управление жизненным циклом объекта (подсчёт ссылок или единоличное владение). **unique_ptr** — единоличное владение, RAII, перемещение; **shared_ptr** — подсчёт ссылок, разделяемое владение; **weak_ptr** — не увеличивает счётчик, нужен для разрыва циклических ссылок.

## 6.6. Out-параметры в функциях и методах

По учебнику (раздел 8.1): при **передаче параметра по значению** функция получает **копию** переменной; изменения внутри функции не влияют на объект вызывающего. При **передаче по ссылке** (или по указателю) функция получает доступ к **самому объекту** вызывающего; изменения внутри функции **изменяют объект снаружи** (побочный эффект). В учебнике 8.1 это демонстрируется примером: в f1 инкремент применялся к копии, в f2 — к самому объекту, так как x была ссылкой на y; в результате f2 изменила значение y вне функции.

**Out-параметры** — использование передачи по **ссылке** или по **указателю** для того, чтобы функция могла **вернуть результат** (или несколько результатов) через параметры: вызывающий передаёт переменную по ссылке, функция записывает в неё значение. Типичная сигнатура: `void compute(int x, int& result)` или `void compute(int x, int* result)`; после вызова `result` содержит выходное значение.

**Альтернативы.** **(1)** Возврат одного значения через **return**; несколько значений — **std::pair**, **std::tuple** или структура по значению. **(2)** **std::optional&lt;T&gt;** — возврат «значение или отсутствие» без отдельного out-параметра для кода ошибки. **(3)** Тип **ResultInfo** (учебник) — инкапсуляция результата и ошибки в одном объекте.

```cpp
// Out-параметр: передача по ссылке
void parse(const std::string& s, int& value, bool& ok) {
    try {
        value = std::stoi(s);
        ok = true;
    } catch (...) {
        ok = false;
    }
}
int val; bool ok;
parse("42", val, ok);  // val и ok заполнены после вызова

// Альтернатива: optional или pair
std::optional<int> parse2(const std::string& s);
auto [val2, ok2] = parse_pair("42");  // structured binding
```

**Итог 6.6.** По учебнику 8.1 передача по ссылке даёт функции доступ к самому объекту вызывающего — этим пользуются для **out-параметров** (возврат результата через параметр). Альтернативы: возврат по значению (pair, tuple, структура), optional, ResultInfo.

## 6.7. Шаблоны и параметрический полиморфизм. Связь с подтипизацией

По учебнику (разделы 6.5.2, 6.5.3, 6.5.4): в C++ сосуществуют два механизма полиморфизма — **параметрический** (шаблоны) и **подтипизация** (наследование, виртуальные функции). Они решают разные задачи и **дополняют друг друга**.

**Параметрический полиморфизм (шаблоны).** По учебнику 6.5.2 **параметрический полиморфизм** позволяет создавать функции и классы, которые могут **работать с любыми типами данных**. В C++ он реализован через **шаблоны (templates)**. Код пишется один раз, а компилятор **автоматически генерирует** конкретные реализации для каждого используемого типа. Разрешение типов и выбор перегрузок выполняются **на этапе компиляции** — **статический полиморфизм**. Зависимость **терм от типа**: значение (функция, класс) зависит от типа-параметра (учебник 6.5.5).

**Подтипизация.** По учебнику 6.5.4 **подтипизация** — отношение между типами, при котором один тип (**подтип**) может использоваться **везде, где ожидается другой тип (надтип)**. В C++ **наследование** — основной способ декларирования подтипизации: если Derived публично наследует Base, то Derived является подтипом Base. По учебнику 6.5.3 **полиморфизм подтипов** позволяет работать с объектами производных классов через **указатели или ссылки на базовый класс**; поведение определяется **фактическим типом объекта во время выполнения** (**динамическое связывание**). В C++ это реализуется **виртуальными методами** (учебник 6.5.3).

**Сравнение.** Шаблоны — **статическая** диспетчеризация, один и тот же код для разных типов без общего базового класса; подтипизация — **динамическая** диспетчеризация, общий интерфейс (базовый класс), выбор метода в runtime. В одном проекте часто используют оба: шаблоны для обобщённых контейнеров и алгоритмов, подтипизацию для полиморфных иерархий и подстановки реализаций.

```cpp
// Шаблон: параметрический полиморфизм, статический (компиляция)
template<typename T>
T add(T a, T b) { return a + b; }
add(1, 2);      // int
add(1.0, 2.0);  // double — генерируется отдельная функция

// Подтипизация: виртуальные методы, динамическая диспетчеризация
struct Base { virtual void foo() = 0; };
struct Derived : Base { void foo() override {} };
Base* p = new Derived();
p->foo();  // вызывается Derived::foo во время выполнения
```

**Итог 6.7.** По учебнику 6.5.2–6.5.4 **шаблоны** реализуют **параметрический полиморфизм** — статический, на этапе компиляции; **подтипизация** (наследование, виртуальные функции) — **полиморфизм подтипов**, динамическая диспетчеризация во время выполнения. Оба механизма дополняют друг друга: шаблоны — обобщённый код без общей базы; подтипизация — общий интерфейс и подстановка реализаций в runtime.

## 6.8. Наследование, ad hoc-полиморфизм и правила переопределения методов и их перегрузок

По учебнику (разделы 6.5.1, 6.5.3): в C++ сосуществуют два механизма выбора реализации по имени метода — **перегрузка (ad hoc полиморфизм)** и **переопределение виртуального метода (полиморфизм подтипов)**. Они различаются **моментом выбора** и **правилами**.

**Перегрузка (ad hoc полиморфизм).** По учебнику 6.5.1 **перегрузка** заключается в определении **нескольких функций или методов с одинаковым именем**, но **разными наборами параметров (сигнатурами)** — отличающихся количеством или типами аргументов. **Выбор конкретной функции происходит статически, во время компиляции.** При перегрузке важно избегать неоднозначностей: компилятор для данного набора аргументов выбирает наиболее подходящую функцию; тип возвращаемого значения **в сигнатуру для разрешения перегрузки не входит** (учебник 6.5.1).

**Переопределение (override).** По учебнику 6.5.3 **полиморфизм подтипов** позволяет работать с объектами производных классов через указатели или ссылки на базовый класс; **поведение определяется фактическим типом объекта во время выполнения** (динамическое связывание). В C++ это реализуется **виртуальными методами**: в производном классе метод **переопределяется** (та же сигнатура, что и в базовом). Ключевое слово **override** в C++11 указывает компилятору, что метод переопределяет виртуальный метод базового класса; компилятор **проверяет сигнатуру** — при несовпадении выдаётся ошибка. В базовых классах с виртуальными функциями **обязателен виртуальный деструктор**, чтобы гарантировать корректное освобождение памяти при удалении объекта производного класса через указатель на базовый (учебник 6.6).

**Сводка.** Перегрузка — **один и тот же идентификатор**, разные **сигнатуры**, выбор на этапе **компиляции**. Override — **та же сигнатура** в производном классе, выбор на этапе **выполнения** по типу объекта.

```cpp
// Перегрузка: выбор при компиляции по типам аргументов
struct Example {
    void Print(int a);
    void Print(double a);
};

// Переопределение: выбор при выполнении по типу объекта
struct Base {
    virtual void Show();
    virtual ~Base() = default;
};
struct Derived : Base {
    void Show() override;  // проверка сигнатуры компилятором
};
```

**Итог 6.8.** По учебнику 6.5.1, 6.5.3 **перегрузка** — ad hoc полиморфизм: одно имя, разные сигнатуры, выбор при компиляции. **Override** — переопределение виртуального метода, выбор при выполнении; ключевое слово **override** проверяет сигнатуру. В базовом классе с виртуальными методами — виртуальный деструктор.

## 6.9. Изменяемые и неизменяемые структуры: сравнительная характеристика

По учебнику (раздел 5.7): **функциональные списки** — структуры данных, которые поддерживают **неизменяемость** и ленивые вычисления. Они **отличаются неизменяемостью** в процессе работы: вместо изменения существующего списка **создаются новые списки**, основанные на старых (с учётом неизменности их элементов) (учебник 5.7).

**Изменяемые структуры (mutable).** Операции **изменяют сам объект** (добавление, удаление, изменение элемента «на месте»). Состояние структуры после операции отличается от состояния до неё. Типично для императивного стиля: вектор, список, множество с методами push_back, insert, erase и т.п. Удобны, когда важна эффективность по памяти (не создаём копии) и последовательность мутаций.

**Неизменяемые структуры (immutable).** Операции **не изменяют исходный объект**; они **возвращают новый объект**, представляющий результат (например, «список с добавленным элементом»), при этом старый объект остаётся доступным и неизменным. По учебнику 5.7 в функциональных списках при добавлении элемента в начало создаётся новый список (1 :: старый_хвост), старый список не меняется. **Неизменяемость** упрощает рассуждения о программе и **параллелизм**: нет разделяемого изменяемого состояния, потоки могут безопасно обращаться к одним и тем же данным без блокировок (учебник 5.7 — функциональные списки эффективны при параллельной обработке).

**Сравнение.** Mutable — меньше копий, привычный императивный стиль; риск гонок и побочных эффектов. Immutable — проще рассуждения и параллелизм; возможны дополнительные затраты на копирование или использование структур с общими частями (persistent data structures).

```cpp
// Mutable: операция изменяет объект
std::vector<int> v = {1, 2, 3};
v.push_back(4);  // v изменился

// Immutable-подобный стиль: возвращаем новый объект
std::vector<int> append(const std::vector<int>& v, int x) {
    std::vector<int> r = v;
    r.push_back(x);
    return r;  // исходный v не изменён
}
```

**Итог 6.9.** По учебнику 5.7 **изменяемые** структуры — операции изменяют объект; **неизменяемые** — операции возвращают новый объект, исходный не меняется. Неизменяемость упрощает рассуждения и параллелизм; изменяемость часто эффективнее по памяти при частых мутациях.

## 6.10. Объекты как глобальные константы: ленивая реализация с помощью мемоизации

По учебнику (раздел 7.3): с логической точки зрения многие объекты — **глобальные константы**: один раз созданы и не меняются. Вопросы на практике: **в какой момент создавать** такие объекты и **как обеспечить**, чтобы они **не создавались повторно**. В качестве решения в учебнике 7.3 предлагается идиома **«объект как глобальная константа»**. **Идея:** с помощью **статического порождающего метода** создавать экземпляр **в момент, когда он впервые запрашивается**, а значение **запоминать в статическом поле**. Это называется **ленивая инициализация с мемоизацией** (учебник 7.3).

**Реализация.** В классе объявляется **статическое поле** (указатель или объект), хранящее единственный экземпляр. **Статический метод** (например, Zero(), Instance(), Get()) при первом вызове создаёт объект (если ещё не создан), сохраняет его в статическом поле и возвращает ссылку (или указатель); при последующих вызовах возвращает уже сохранённое значение — **мемоизация**. По учебнику 7.3 пример: класс Vector3 со статическим полем m_zero и методом Zero(): при первом обращении к Zero() создаётся Vector3(0,0,0), записывается в m_zero, возвращается *m_zero; при следующих вызовах возвращается тот же объект.

**Статическая локальная переменная (C++).** Альтернатива — **статическая локальная переменная** внутри функции-аксессора: при первом входе в функцию она инициализируется (например, конструктором или фабричной функцией), при последующих входах используется уже созданный объект. В **C++11** инициализация статической локальной переменной при первом проходе **потокобезопасна** (magic statics): компилятор генерирует код с однократной инициализацией под блокировкой.

```cpp
// По учебнику 7.3: ленивая инициализация с мемоизацией
class Vector3 {
    static Vector3* m_zero;
public:
    static const Vector3& Zero() {
        if (!m_zero) m_zero = new Vector3(0.0f, 0.0f, 0.0f);
        return *m_zero;
    }
private:
    float x, y, z;
    Vector3(float x, float y, float z) : x(x), y(y), z(z) {}
};

// Вариант со статической локальной переменной (C++11 — потокобезопасно)
const Vector3& Zero() {
    static const Vector3 zero(0.0f, 0.0f, 0.0f);
    return zero;
}
```

**Итог 6.10.** По учебнику 7.3 **объект как глобальная константа** реализуется **ленивой инициализацией с мемоизацией**: статический порождающий метод создаёт экземпляр при первом запросе и запоминает его в статическом поле (или в статической локальной переменной). В C++11 инициализация такой статической локальной переменной **потокобезопасна**.

## 6.11. Контроль и управление памятью

По учебнику (разделы 8.3, 8.4): управление памятью и ресурсами в C++ опирается на **семантику копирования** и **семантику перемещения**, а также на **порядок вызова конструкторов и деструкторов**.

**Копирование.** По учебнику 8.3 при передаче параметра по значению функция получает **копию** переменной; для составных типов (классов, структур) копия создаётся с помощью **конструктора копирования**: `T(const T& object)`. Конструктор копирования принимает объект и строит новый объект на его основе. Если он не указан явно, создаётся **конструктор по умолчанию**, который копирует каждое поле; при полях-указателях получается **поверхностное копирование** — оба объекта указывают на одни и те же данные, что может быть нежелательно (учебник 8.3, пример DynamicArray). **Оператор присваивания** `operator=(const T&)` должен корректно обрабатывать копирование и **самоприсваивание**.

**Перемещение.** По учебнику 8.4 **конструктор перемещения** `T(T&& other)` используется для **передачи ресурсов** от одного объекта к другому **без копирования**: он забирает владение у временного объекта. Параметр **T&&** — **rvalue-ссылка** на объект типа T. Конструктор перемещения вызывается, когда новый объект создаётся из **временного (rvalue)** — например, при возврате из функции или при **std::move**. По учебнику 8.4 **std::move** не перемещает данные сама, а лишь **преобразует объект в rvalue**, позволяя компилятору вызвать конструктор перемещения вместо копирования. **lvalue** — выражение, имеющее адрес (переменная, результат операции, возвращающая ссылку); **rvalue** — временное значение (литерал, результат функции по значению, результат std::move).

**Виртуальный деструктор.** В базовых классах с виртуальными функциями **обязателен виртуальный деструктор**, чтобы при удалении объекта производного класса через указатель на базовый вызывался деструктор производного класса и ресурсы освобождались корректно (учебник 6.6).

**Порядок вызова.** Конструктор: сначала **базовые подобъекты** (в порядке объявления), затем **члены** (в порядке объявления), затем тело конструктора. Деструктор — **в обратном порядке**: тело деструктора, затем члены, затем базы.

**RAII.** Принцип **«захват ресурса при инициализации»** (Resource Acquisition Is Initialization): ресурс (память, файл, сокет) привязывается к объекту; конструктор захватывает ресурс, деструктор освобождает. Гарантирует освобождение при выходе из области видимости (в т.ч. при исключениях). Умные указатели (unique_ptr, shared_ptr) — типичная реализация RAII для памяти.

**Правило трёх и правило пяти.** Если класс управляет ресурсом (память, дескриптор), обычно нужно определить несколько связанных операций. **Правило трёх** (до C++11): деструктор, копирующий конструктор, копирующее присваивание — при наличии одного обычно нужны все три. **Правило пяти** (C++11): добавляются конструктор перемещения и оператор присваивания перемещением. Если не определить их, по умолчанию сгенерируются «поверхностные» версии, что при указателях приведёт к двойному освобождению или утечкам.

```cpp
// По учебнику 8.3, 8.4 — пример правила пяти
struct Example {
    int* data;
    Example(const Example& other) {  // копирование: глубокая копия
        data = other.data ? new int(*other.data) : nullptr;
    }
    Example& operator=(const Example& other) {
        if (this != &other) {
            delete data;
            data = other.data ? new int(*other.data) : nullptr;
        }
        return *this;
    }
    Example(Example&& other) noexcept {  // перемещение: забираем владение
        data = other.data;
        other.data = nullptr;
    }
    Example& operator=(Example&& other) noexcept {
        if (this != &other) {
            delete data;
            data = other.data;
            other.data = nullptr;
        }
        return *this;
    }
    ~Example() { delete data; }
};
// std::move(x) — приведение x к rvalue, вызов move-конструктора/присваивания
```

**Итог 6.11.** По учебнику 8.3, 8.4 **копирование** — конструктор копирования и оператор присваивания; при указателях нужна глубокая копия или явное управление. **Перемещение** — T(T&&), std::move, rvalue; передача ресурсов без копирования. **RAII** — ресурс привязан к объекту, деструктор освобождает. **Правило пяти** — при управлении ресурсом определять деструктор, копирование (конструктор + присваивание), перемещение (конструктор + присваивание). **Виртуальный деструктор** обязателен при полиморфном delete. Порядок вызова: базы → члены → тело конструктора; уничтожение в обратном порядке.

## 6.12. Абстрагирование зависимостей и разделение ответственностей (на примере Sequence)

По учебнику (раздел 3.2.4, 3.2.5): последовательность можно реализовать **на базе связного списка** или **на базе массива**; каждая реализация имеет свои преимущества. Целесообразно создать **два конкретных наследника**: ArraySequence (динамический массив) и ListSequence (связный список). При этом предпочтительным является подход **на основе инкапсуляции**: функциональность конкретной структуры **делегируется внутреннему полю** нужного типа (ArraySequence содержит приватное поле типа DynamicArray, ListSequence — поле типа LinkedList). Такой подход **позволяет работать с последовательностями, не вдаваясь в детали реализации**, и сохраняет возможность выбора наиболее эффективной структуры для конкретной задачи; например, при реализации алгоритма сортировки можно использовать **параметр типа Sequence** и сравнивать производительность массива и списка (учебник 3.2.4, 3.2.5).

**Абстрагирование зависимостей.** Код, использующий последовательность, зависит от **интерфейса** (абстрактный тип Sequence с методами hasNext, next, get и т.п.), а **не от конкретной реализации** (ArraySequence, ListSequence, FileSequence). **Разделение ответственностей:** алгоритм обработки (process) отвечает только за обход и использование элементов; источник данных (массив, файл, сеть) инкапсулирован в конкретной реализации Sequence. Конкретная реализация передаётся снаружи (внедрение зависимости); код process **не знает**, массив это или файл.

```cpp
// Зависимость от интерфейса, а не от реализации (по учебнику 3.2.4)
struct ISequence {
    virtual bool hasNext() = 0;
    virtual int next() = 0;
    virtual ~ISequence() = default;
};
void process(ISequence& seq) {
    while (seq.hasNext()) use(seq.next());
}
// Вызов: process(arraySeq); или process(fileSeq); — process не различает
```

**Итог 6.12.** По учебнику 3.2.4, 3.2.5 **абстрагирование зависимостей** — зависимость от интерфейса (Sequence), а не от реализации (ArraySequence, ListSequence). **Разделение ответственностей:** алгоритм работает с интерфейсом; конкретная реализация передаётся снаружи. Пример: process(ISequence&) не знает источник данных.

## 6.13. Инкапсуляция и изоляция. Связь с friend-классами

По учебнику: **инкапсуляция** — принцип **сокрытия реализации**; данные и детали реализации сосредоточиваются внутри объекта, доступ к ним — через заданный интерфейс (публичные методы). В C++ для этого служат модификаторы доступа **private** и **protected**: закрытые члены недоступны снаружи класса. **Изоляция** — изменения внутри класса (замена полей, алгоритмов) не должны ломать внешний код, пока интерфейс сохраняется; инкапсуляция уменьшает связность и упрощает сопровождение (учебник: инкапсуляция в ООП, API).

**friend.** В C++ ключевое слово **friend** даёт выбранной функции или классу **доступ к private- и protected-членам** данного класса, обходя обычные правила доступа. **Использование:** **(1)** **Симметричные операторы** — по учебнику 6.7 для выражений вида `k * v` (скаляр × вектор) пишут внешние перегрузки с **friend** или шаблонные, чтобы функция имела доступ к приватным полям вектора при необходимости. **(2)** **Тесты** — тестовый класс или функция может быть объявлена friend для доступа к внутреннему состоянию при unit-тестировании. **(3)** **Связанные компоненты** — фабрики, вспомогательные классы, которым нужен доступ к внутренностям. **Недостаток:** friend **ослабляет инкапсуляцию** — число мест, имеющих доступ к закрытым членам, увеличивается; изменения в представлении класса могут потребовать правок во всех friend-объектах.

```cpp
class Vector {
    double x, y;
public:
    friend Vector operator*(double k, const Vector& v);  // доступ к x, y
};
Vector operator*(double k, const Vector& v) {
    return Vector(k * v.x, k * v.y);
}
```

**Итог 6.13.** **Инкапсуляция** — сокрытие реализации (private/protected), доступ через интерфейс; **изоляция** — ограничение влияния изменений внутри класса. **friend** даёт выбранной функции или классу доступ к private/protected; используется для симметричных операторов (учебник 6.7), тестов, связанных компонентов; ослабляет инкапсуляцию.

## 6.14. Понятие объектной модели. Пример

По учебнику (раздел 6.6): **объектная модель** — **представление рассматриваемой системы в терминах объектов и отношений между ними**. Этот подход является отправной точкой для структурирования сложных систем. В программе мы оперируем не самими предметами предметной области, а их **именами** (образами в языке). Переход к ООП синтаксически упростил работу с объектной моделью: если раньше для доступа к атрибуту объекта требовался вызов отдельной функции, то в ООП введён прямой синтаксис (**объект.атрибут**, **объект.метод**), что создало прочную ассоциацию между объектом и его членами (атрибутами и методами) (учебник 6.6).

**Объектная модель на уровне реализации (C++).** На уровне языка и компилятора объектная модель задаёт **правила размещения объектов в памяти**. **Члены класса** (поля) располагаются **последовательно** в порядке объявления; компилятор может добавлять **выравнивание** (padding) для эффективного доступа. При **наследовании** сначала идёт подобъект базового класса, затем члены производного. При **виртуальных функциях** компилятор добавляет в объект **скрытый указатель (vptr)** на таблицу виртуальных функций (**vtable**); обычно vptr размещается в начале объекта (перед полями). **Вызов виртуальной функции** выполняется через vptr: по индексу метода в vtable выбирается указатель на функцию и выполняется косвенный вызов — поэтому выбор реализации происходит **во время выполнения** (динамическая диспетчеризация).

**Пример.** Класс с виртуальным методом foo() и полями int a, int b: в памяти сначала vptr (указатель на vtable), затем a, затем b (с учётом выравнивания). Вызов p->foo() превращается в вызов (*p->vptr[index_foo])(p).

```cpp
// Упрощённо: объект в памяти
struct Base {
    virtual void foo() {}
    int a, b;
};
// Layout (типично): [vptr][a][b]; vptr указывает на vtable с адресами виртуальных функций
```

**Иллюстрация объектной модели (объекты и отношения) и размещения в памяти (C++):**

```
Объектная модель (предметная область):
  ┌─────────┐        ┌─────────┐
  │ Объект A│───────▶│Объект B │     Отношения: A ссылается на B
  │ атрибут1│        │ атрибут │     В программе: objA.method(), objB.attr
  │ атрибут2│        └─────────┘
  └─────────┘

Размещение объекта в памяти (C++, класс с виртуальными методами):
  Объект Base:                    vtable (таблица виртуальных функций):
  ┌──────────────┐                ┌─────────────────┐
  │ vptr ────────┼───────────────▶│ [0] &Base::foo  │
  ├──────────────┤                │ [1] &Base::bar  │  (если есть)
  │ a (int)      │                └─────────────────┘
  ├──────────────┤
  │ b (int)      │     Вызов p->foo():  (*p->vptr[0])(p)
  └──────────────┘     → по vptr находим vtable, по индексу — адрес foo(), вызываем
```

**Итог 6.14.** По учебнику 6.6 **объектная модель** — представление системы в терминах объектов и отношений; в ООП — прямой синтаксис доступа к атрибутам и методам. На уровне реализации C++: члены размещаются последовательно, выравнивание; при виртуальных функциях — vptr и vtable; вызов виртуального метода — через vptr по индексу.

## 6.15. Понятие атрибута объекта. Реализация концепции атрибута средствами ООП

По учебнику: **атрибут** (от лат. attributio — «приписывание») — **признак, приписываемое качество или свойство** объекта. Человек воспринимает окружающий мир через объекты и их атрибуты; реляционная модель и ООП опираются на представление данных в виде **объектов и их атрибутов**. В ООП объекты обладают **атрибутами и методами**; состояние и поведение инкапсулированы в целое (учебник: введение в ООП). **Внешняя спецификация** объекта включает **атрибуты** — характеристики объекта, которые могут быть доступны или изменены (учебник: внешняя спецификация АТД).

**Реализация атрибута средствами ООП.** По учебнику 6.6 при написании классов **хранимым атрибутам** предметной области сопоставляются **приватные поля** с доступом через **публичные методы (геттеры)**. По учебнику (инкапсуляция): для решения проблем согласованности и контроля доступа **поля делают приватными**, а доступ к ним осуществляют **через геттеры и сеттеры**, обеспечивающие контроль за изменениями. Таким образом, **атрибут** как именованное свойство с типом реализуется в C++ как **поле (данные)** плюс **геттер** (метод чтения, getX()) и при необходимости **сеттер** (метод записи, setX()) — публичный интерфейс к приватному хранению.

**Зачем геттер/сеттер.** Единая точка доступа позволяет проверять и валидировать значения при записи, вычислять производные атрибуты при чтении (например, полное имя из имени и фамилии), менять внутреннее представление без изменения внешнего интерфейса.

```cpp
// Атрибут: именованное свойство с типом; реализация — поле + геттер + сеттер
class Person {
    std::string firstName_;
    std::string lastName_;
public:
    const std::string& getFirstName() const { return firstName_; }
    void setFirstName(const std::string& s) { firstName_ = s; }
    std::string getFullName() const { return firstName_ + " " + lastName_; }  // вычисляемый атрибут
};
```

**Итог 6.15.** По учебнику **атрибут** — признак, приписываемое свойство объекта; во внешней спецификации — характеристика, доступная или изменяемая. **Реализация в ООП:** приватное поле + геттер (и при необходимости сеттер); по учебнику 6.6 атрибутам сопоставляются приватные поля с доступом через публичные методы (геттеры).

## 6.16. Подтипизация и наследование. Изменение внешней спецификации при подтипизации

По учебнику (раздел 6.5.4): **подтипизация** — это отношение между типами, при котором **один тип (подтип) может использоваться везде, где ожидается другой тип (надтип)**. В C++ **наследование** является основным способом декларирования отношения подтипизации. Если класс Derived **публично наследуется** от Base, то Derived является **подтипом** Base. Это означает, что **всё, что справедливо для Base, справедливо и для Derived**, но не наоборот. Отношение подтипизации рефлексивно, транзитивно и антисимметрично — образует частичный порядок на множестве типов (учебник 6.5.4).

**Изменение внешней спецификации при подтипизации.** Внешняя спецификация метода задаётся **предусловиями** (что должно выполняться до вызова) и **постусловиями** (что гарантируется после вызова). **Принцип подстановки Барбары Лисков (LSP):** объекты подтипа должны быть **заменяемы** объектами надтипа без нарушения корректности программы. Отсюда правила при переопределении методов в производном классе: **нельзя сужать предусловия** (требовать от вызывающего больше, чем базовый метод — иначе код, рассчитанный на Base, может нарушить предусловия Derived); **нельзя ослаблять постусловия** (гарантировать меньше, чем базовый метод — иначе контракт надтипа перестаёт выполняться). При наследовании **можно только расширять гарантии**: ослаблять предусловия (принимать больше входов) и усиливать постусловия (гарантировать больше). Тогда любой код, работающий с Base, остаётся корректным при подстановке Derived.

```cpp
// По учебнику 6.5.4: Derived — подтип Base, везде, где ожидается Base, можно подставить Derived
struct Base {
    virtual void process(int x) {}  // предусловие: x >= 0; постусловие: состояние обновлено
    virtual ~Base() = default;
};
struct Derived : public Base {
    void process(int x) override {}  // LSP: предусловие не сужено, постусловие не ослаблено
};
void use(Base& b) { b.process(1); }
// Derived d; use(d);  — LSP: можно передать Derived вместо Base
```

**Итог 6.16.** По учебнику 6.5.4 **подтипизация** — подтип используется везде, где ожидается надтип; в C++ задаётся публичным наследованием. При переопределении методов **LSP**: не сужать предусловия, не ослаблять постусловия; при наследовании можно только расширять гарантии.

## 6.17. Подтипизация и наследование. Проблемы множественного наследования. Ромбовидное наследование

По учебнику (раздел 6.4): особенностью наследования в C++ является **возможность множественного наследования** — класс может **наследовать функциональность от нескольких классов одновременно**. Это позволяет комбинировать разные типы поведения и состояния в одном подклассе. Для разрешения ситуаций, в которых у разных родительских классов присутствуют **одинаковые методы**, используются механизмы **явного указания на родительский класс** с помощью операторов разрешения области видимости (Base::method()) (учебник 6.4).

**Множественное наследование.** В объекте производного класса создаётся **несколько подобъектов** — по одному на каждую базу. Если в двух базах есть член с одним именем, обращение к нему **неоднозначно** — нужно указывать класс: `d.B::foo()` или `d.C::foo()`.

**Ромбовидное наследование (diamond).** Иерархия: A ← B, A ← C, B и C ← D (D наследует и B, и C, а B и C оба наследуют A). **Без виртуального наследования** в объекте D оказывается **два подобъекта A** (один через B, один через C). Обращение к члену A из D снова неоднозначно; вызов метода A может быть неоднозначным. **Виртуальное наследование** (virtual public A в B и C): в D остаётся **один подобъект A**, общий для B и C. Конструктор **виртуального базового класса A** вызывается **из самого производного класса D** (а не из B и C), чтобы A был создан один раз; порядок вызова конструкторов при виртуальном наследовании регулируется стандартом.

```cpp
// Ромбовидное наследование
struct A { int x; };
struct B : virtual A {};
struct C : virtual A {};
struct D : B, C {};  // один подобъект A

// Без virtual: struct B : A {}; struct C : A {}; struct D : B, C {};
// в D два подобъекта A, обращение к x неоднозначно: d.x — ошибка
```

**Итог 6.17.** По учебнику 6.4 **множественное наследование** в C++ — несколько базовых подобъектов; при общем члене — неоднозначность, разрешается через Base::member. **Ромбовидное наследование** (A←B, A←C, B,C←D): без **virtual** — два подобъекта A в D; с **виртуальным наследованием** — один подобъект A, конструктор A вызывается из D.
